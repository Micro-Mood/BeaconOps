## 3. 核心服务实现要点

### 3.1 设备注册表(DeviceRegistry)
- 启动加载所有设备到 Redis 内存
- 监听 EMQX webhook(`client_connected` / `client_disconnected`)
- 维护 `is_online` 实时镜像
- **白名单热缓存**:`trusted:{uuid} → 1/0` 写入 Redis,EMQX 鉴权 webhook 命中 Redis(< 1ms);`POST /devices/:id/trust` 与 `revoke` 同时双写 PG + Redis,撤销立即生效
- 提供查询:online 设备数、按角色聚合、按部门聚合、按白名单状态聚合

### 3.2 消息调度器(MessageDispatcher)
```python
async def dispatch(msg: MessageCreate, operator: User):
    # 1. RBAC 校验
    rbac.check(operator, "send_message", target=msg.target)
    # 2. 解析目标 → 实际 device_id 列表
    device_ids = await resolve_targets(msg.target_type, msg.target_value)
    # 3. 持久化
    msg_id = await save_message(msg, operator)
    # 4. TTS 分流决策(节省云 TTS 额度)
    if msg.audio and msg.audio.mode == "tts_stream":
        # 广播 / 多点 / 重复文本 → 服务端预合成一次,下发 url(端侧不调云 TTS)
        # 判断:接收者 > 1 OR voice_text 哈希命中全局缓存
        cache_key = sha256(msg.audio.text + msg.audio.voice)
        if len(device_ids) > 1 or await tts_cache.exists(cache_key):
            url = await tts_cache.get_or_synth(cache_key, msg.audio)
            msg.audio.mode = "preset_url"
            msg.audio.url  = url
            msg.audio.text = None        # 不再带文本,端侧走 url
        # else: 1v1 个性化文本 → 保留 text,端侧以凭据调云 TTS
    # 5. 立即 / 定时
    if msg.schedule_at:
        await scheduler.add_job(fire_message, run_date=msg.schedule_at, args=[msg_id])
    else:
        await mqtt_publisher.send_to_devices(device_ids, msg.payload)
    # 6. 启动 escalation 看护(若 require_ack)
    if msg.require_ack:
        asyncio.create_task(watch_escalation(msg_id, msg.timeout_ms))
    return msg_id
```

**TTS 分流决策总结**:
| 场景 | 合成者 | 传输 |
|---|---|---|
| 广播 / 多设备接收 | **服务端一次合成**后 CDN/Redis 缓存 | 下发 url,设备 HTTP GET PCM/Opus |
| 重复文本哈希命中缓存 | 服务端复用 | 下发 url |
| 1v1 个性化文本 | **端侧**调云 TTS | 只传文本 |

> 避免 N 台设备同一句各调 1 次 → 云 TTS 额度 N 倍泄露。

### 3.3 MQTT Bridge
- 单进程一个 `asyncio-mqtt` 客户端,作为系统超级用户连接 broker
- 订阅 `device/+/ack`、`device/+/telemetry`、`device/+/status`、`device/+/shake`
- 收到消息:`schemas` 校验 → 入库 → 触发 WS 广播
- 发送下行:统一通过 publisher 接口,记 audit log

### 3.4 TTS 服务(凭据分发)

**架构变更**:TTS 主要在**端侧调用**(详见 02 §6.3)。服务端**不预烧凭据**,设备运行期从服务端拉取,凭据不落盘。

**凭据分发接口**(唯一 TTS 相关接口):
```
GET /api/v1/device/tts/credential
Headers:
  X-Device-Id: E1024
  X-Timestamp: <unix_seconds>
  X-Nonce: <16 hex>
  X-Signature: HMAC-SHA256(device_secret, f"{device_id}|{ts}|{nonce}")
→ 200
{
  "apikey": "sk-xxx",
  "tts_model": "azure:zh-CN-XiaoxiaoNeural",
  "endpoint": "https://tts.example.com/v1/synth",
  "expires_at": 1747203600
}
```

服务端职责:
- 验 HMAC 签名(`DEVICE_SECRET` 按 `devices.batch_id` 从 vault 取出;同批共享)
- 验时间窗口 ±60s + nonce 在 Redis 反重放(TTL 120s)
- 从重点企业账号 vault 读出 apikey + 动态生成凭据(可是企业主账号 key,也可是按设备生成的子凭据)
- TTL 默认 1 小时,设备提前 5 分钟调起续签
- 限频:同一 device_id 60s 内限 1 次,避免被刷
- 记 audit:设备获取凭据 是应该被全部记录的敏感操作

**提供商切换**:后台调整 vault 中的 provider/model 即可,下一次设备拉取凭据时生效。

**变更推送**(可选):服务端下发 `cmd { type: "tts_credential_invalidate" }` → 设备丢弃现凭据 → 重新拉取。

**安全责任划分**:
- 服务端:按 device_id 设置 TTS 调用额度(限频 + 月上限),key 被滥用可仅限某设备黑名单
- 端侧:HMAC 签名(§6.3),DRAM only,memset 清零
- 企业:vault 里的主账号 key 可随时轮换,设备无感

### 3.5 LLM Gateway + MCP
- `/llm/sessions/:sid/messages`:接收用户文本 → 启动 chat loop
- chat loop:
  ```
  while not done:
      response = await llm.chat_stream(msgs, tools=mcp.list_tools())
      for chunk in response:
          if chunk.is_text: yield SSE "text"
          if chunk.is_tool_call:
              result = await mcp.call(chunk.name, chunk.args, ctx=operator)
              yield SSE "tool_call" + "tool_result"
              msgs.append(...)
      if no more tool_calls: done = True
  ```
- 上下文压缩:消息 > 20 轮时,后台用 LLM 摘要前 N 轮 → 写入 `llm_sessions.summary`,新会话用 summary 替代

**MCP 工具集**(摘要,详见 `项目总体方案.md` §8.2):
| 工具 | 副作用 | 是否敏感 |
|---|---|---|
| `list_devices(filter)` | 只读 | 否 |
| `get_device_profile(device_id)` | 只读 → today 指标 + 7d 滑动均值 + last_active_ts + response_rate_7d + delivery_rate_7d | 否 |
| `get_recent_messages(device_id, limit)` | 只读 | 否 |
| `send_message(target, payload)` | 写,scope=user/role | **scope=role/dept/all 时敏感** |
| `broadcast_emergency(scope, text)` | 写,广播 | **是** |
| `set_device_alias / volume` | 写,单设备 | 否 |
| `delete_device(device_id)` | 写,毁灭性 | **是** |

**敏感操作二次确认机制**(防 LLM prompt injection 直接执行毁灭操作):
```
LLM tool_call(sensitive=True) 
  → MCP Guard 不直接执行
  → 返回 {"status":"need_confirm","confirm_token":"<jwt 60s 一次性>","plan":{...}}
  → 前端 ToolCallCard 渲染:plan 详情 + [确认执行] / [取消] 按钮
  → 用户点击确认 → 前端 POST /llm/confirm {token} → 真正执行
  → 取消 / 60s 未确认 → token 作废
audit_log 完整记录两阶段(propose / confirm),操作人 = 当前登录人,非 LLM
```
- 敏感判定:工具元数据 `sensitive=True` 或参数命中危险范围(scope in role/dept/all、delete_*)
- token 由服务端签名,绑定 `(operator_id, tool_name, args_hash, exp)`,防重放

### 3.6 Behavior Decider(创新点 3)
- 后台定时任务:每 5 分钟扫描所有在线设备的最近遥测 + `device_profiles`
- 触发器规则(粗筛):
  - static_secs > 1800(30min)
  - emergency 消息 displayed 但 5min 未 acknowledged
  - 跌倒事件
  - **个人化阈值**:某设备习惯 static_secs > avg_static_7d × 1.3 才警示(避免却合靶)
- LLM 上下文会注入该设备的画像:
  ```json
  {"profile": {"avg_static_7d": 1500, "response_rate_7d": 0.92, "delivery_rate_7d": 0.99, "steps_today": 3200, ...}}
  ```
- 命中后 → 调用 LLM(系统 prompt 内置策略)→ 决定是否生成干预消息 → 经 dispatcher 下发

### 3.7 Scheduler(定时消息)
- 用 APScheduler + AsyncIOScheduler + SQLAlchemyJobStore(持久化)

### 3.8 Escalation 看护
- emergency 消息 displayed 后 timeout 内未 acknowledged →

  - 记录 audit + WS 实时通知
- 非 emergency 消息:仅记录 expired/未 ack,不升级

### 3.10 设备画像更新器(ProfileUpdater)
**写入路径(全部先走 Redis,60s 批量 flush 到 PG)**:
```python
# 热路径(高频)—— 只动 Redis hash,不写 PG
async def on_ack(ack: AckMsg):
    await save_ack_row_async(ack)               # message_acks(可走 batch insert)
    if ack.type == "acknowledged":
        field = "manual" if ack.via in ("shake","button") else "auto"
        await redis.hincrby(f"profile:{ack.device_id}", f"{field}_ack_today", 1)
    await redis.hset(f"profile:{ack.device_id}", "last_active_ts", now_ts())

async def on_telemetry(t: Telemetry):
    # 去重:UNIQUE(device_id, seq) 冲突 → 丢弃(重复上报,不计额外 profile)
    inserted = await pg.insert_telemetry_ignore_dup(t)
    if not inserted:
        return
    # 端侧上报的是增量,服务端累加到 Redis hash
    await redis.hincrby(f"profile:{t.device_id}", "static_total_today", t.static_secs_inc)
    await redis.hincrby(f"profile:{t.device_id}", "steps_today",        t.steps_inc)
    await redis.hincrby(f"profile:{t.device_id}", "shake_count_today",  t.shake_inc)
    await redis.hset   (f"profile:{t.device_id}", "longest_static_today",
                        max(redis.hget(...,"longest_static_today") or 0, t.longest_static_seen))
    # 发 ACK 主题 device/{id}/ack telemetry_persisted+seq → 端侧原子清 NVS profile_delta + seq++
    await mqtt_publisher.publish(f"device/{t.device_id}/ack",
        {"type":"telemetry_persisted","seq":t.seq})

# 冷路径:60s 定时 flush
async def flush_profiles_to_pg():
    for device_id in await redis.scan("profile:*"):
        h = await redis.hgetall(device_id)
        await pg.upsert("device_profiles", h)

# 服务端跨日 reset(不依赖端侧时钟)
# APScheduler cron 00:00 Asia/Shanghai 每日触发
async def daily_rollup():
    # 1. 读取 device_profiles 当日指标 → 插入 daily_profile_history
    # 2. 重算滑动七日均值 → avg_static_7d / response_rate_7d / delivery_rate_7d
    # 3. 重置当日字段(date_yyyymmdd, *_today=0)
    # 4. 同步清 Redis profile:* 当日计数
```

**两个指标分开不要混淆**:
```
响应率 response_rate = manual_ack / (manual_ack + auto_ack)
     衡量“看到了也肯主动确认”的习惯,分母不含 expired
送达率 delivery_rate = (manual_ack + auto_ack) / msg_received
     衡量设备 + 网络是否能把消息送上屏
```

**提供查询 API**:
- `GET /devices/{id}/profile` → 实时画像(先试 Redis,未命中读 PG)
- `GET /devices/{id}/profile/history?days=30` → 历史趋势(前端 ECharts)

## 4. 安全实现细节

| 项 | 实现 |
|---|---|
| 密码 | argon2-cffi |
| JWT | HS256,access 30min,refresh 7d |
| 密钥轮换 | 配置 KMS(MVP 用 env) |
| HMAC 设备鉴权 | EMQX 用 Webhook 鉴权钩子,转 FastAPI `/auth/device`;校验三层:① `devices.is_trusted = true` 白名单;② 按 `batch_id` 从 vault 取共享 secret 验 HMAC 签名;③ timestamp ±5min + Redis nonce(TTL 300s)防回放 |
| TLS | 单向认证(验服务器证书);公网 Let's Encrypt(设备烧 ISRG Root X1 公开 PEM,免申请),内网自签 Root CA(10 年);设备只烧 Root CA,叶子证书可热轮换 |
| 设备配额 | 按 `device_id`(UUID)限频(连接/分钟、消息/小时、TTS 凭据请求/分钟);共享 secret 泄露场景下管理员可一键 `revoke`(`is_trusted=false`)立即作废 |
| CSRF | API 使用 Bearer,无 cookie 不需要 |
| 限流 | slowapi + Redis 滑窗 |
| RBAC | 装饰器 `@require_perm("send_message")` + 数据级过滤 |
| LLM 越权 | MCP Guard(见创新点 2) |
| SQL 注入 | SQLAlchemy 参数化,不拼字符串 |
| 输入校验 | Pydantic Schema 强类型 + 自定义 validator |
| 日志脱敏 | 中间件过滤 password/token 字段 |

### 8.1 SSE 封装(LLM 流式)
```ts
// useSSE.ts
export async function streamLLM(sessionId: string, content: string,
                                 onEvent: (e: LLMEvent) => void) {
  const resp = await fetch(`/api/v1/llm/sessions/${sessionId}/messages`, {
    method: 'POST',
    headers: { 'Authorization': `Bearer ${token}`, 'Content-Type': 'application/json' },
    body: JSON.stringify({ content }),
  });
  const reader = resp.body!.getReader();
  const decoder = new TextDecoder();
  let buf = '';
  while (true) {
    const { value, done } = await reader.read();
    if (done) break;
    buf += decoder.decode(value, { stream: true });
    const lines = buf.split('\n\n'); buf = lines.pop()!;
    for (const line of lines) {
      if (line.startsWith('data: ')) {
        onEvent(JSON.parse(line.slice(6)));
      }
    }
  }
}
```

### 8.2 WebSocket 事件总线
```ts
// stores/events.ts
export const useEventsStore = defineStore('events', () => {
  const ws = ref<WebSocket | null>(null);
  const handlers = new Map<string, Set<Function>>();
  function connect() {
    ws.value = new WebSocket(`${WS_BASE}/ws/events?token=${token}`);
    ws.value.onmessage = (e) => {
      const evt = JSON.parse(e.data);
      handlers.get(evt.type)?.forEach(fn => fn(evt));
      handlers.get('*')?.forEach(fn => fn(evt));
    };
    ws.value.onclose = () => setTimeout(connect, 2000);  // 自动重连
  }
  function on(type: string, fn: Function) { /* ... */ }
  return { connect, on };
});
```
