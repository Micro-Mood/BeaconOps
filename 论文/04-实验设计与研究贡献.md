## 1. 实验平台与数据集

### 1.1 硬件实验台(Testbed)
- **设备**:**自研 ESP32-C3 智能指挥工牌 × 1 台**(序列号 `E0001`)
- **网络**:企业级 Wi-Fi(Cisco AP)+ EMQX 5.x broker(单节点 4C8G,放在同一二层网内排除外网抖动)
- **后端**:1 台 Ubuntu 22.04 服务器(8C16G + NVMe),运行 FastAPI / PG16+Timescale / Redis 7
- **流量发生器**:`tools/load_gen.py` — 每条消息注入精确 server_emit 时刻:
  ```python
  async def emit_msg(client, device_id, level):
      payload = {
          "id": str(uuid4()),
          "level": level,              # info/notice/warn/emergency
          "display": {"title": f"T{level}", "text": "x"*64},
          "ttl": TTL_BY_LEVEL[level],  # info=30s warn=120s emerg=0
          "ts": int(time.time()),
          "_inject_ts": time.time()    # 双写:load_gen 本地精确 emit 时刻
      }
      await client.publish(f"device/{device_id}/cmd", json.dumps(payload), qos=1)
      await pg.execute("INSERT INTO msgs(id,inject_ts,level) VALUES($1,$2,$3)",
                       payload["id"], payload["_inject_ts"], level)
  ```
- **被动探针**:端侧 `sniffer_task`(独立 FreeRTOS 任务,优先级最高)在每个关键事件写 UART:
  ```
  # 每行 CSV,字段固定:
  PROBE,<event>,<msg_id_8b>,<boot_ms>,<wall_ts_ms>,<q_size>,<level>
  # event 取值:
  #   RECV      — MQTT on_message 回调(DMA 完成后)
  #   ENQUEUE   — 通过去重、写入 queue[]
  #   DEDUP_HIT — LRU 命中,丢弃
  #   DISPLAY   — LVGL lv_scr_load_anim 触发前
  #   ACK_SENT  — MQTT publish acknowledged 前
  #   PREEMPT   — 新消息抢占当前显示项
  #   EXPIRE    — TTL 到期弹出
  #   STARVE    — 消息 TTL 到期但从未 DISPLAY(饥饿事件)
  ```
  `tools/probe_collector.py` 通过 USB-UART 实时读取,写入 `data/probe_{device}_{run}.csv`
- **时钟对齐**:实验前所有设备 SNTP 同步本地 NTP,误差 ≤ 5ms;端侧每条探针双写 `wall_ts`(SNTP 秒级)+ `boot_ms`(esp_timer 毫秒级),后处理脚本自动检测 SNTP 漂移并修正

## 2. PA-MQS 实验(对应第 6 章 "端侧调度算法")

### 2.1 假设(可证伪表述)
- **H1**:在中-高负载(λ ≥ 0.5 msg/s)且含 EMERG 的混合流下,PA-MQS 的 **EMERG p95 端到端延迟** 比"严格 FIFO"低 ≥ 50%
- **H2**:在长程压测(8h)中,PA-MQS 的 **低优先级饥饿率(从未上屏率)** ≤ 1%,显著低于"严格优先级队列"(预期 ≥ 10%)
- **H3**:`min_display_ms = 2000` 抢占保护使 EMERG-vs-EMERG 抢占带来的"用户错过率"(显示时间 < 500ms 即被替换)从 X% 降到 ≈ 0%

### 2.2 基线实现要点(必须对比 5 条,论文里画 6 条曲线)

| 编号 | 基线 | 关键差异 | 实现方式 |
|---|---|---|---|
| B0 | 严格 FIFO | 无优先级 | `queue[]` 纯环形,入队不排序;`msg_task` 永远取 head |
| B1 | 严格优先级队列(static priority, no aging) | 高优先长期占满 → 低优先饥饿 | 入队时插入到同级末尾;无 aging_bonus;无 min_display |
| B2 | 加权公平队列 WFQ | level 权重比 1/2/4/8 | 维护 4 个 FIFO bucket;每次调度按权重轮询 |
| B3 | EDF(Earliest Deadline First) | 按 expire_ts 排 | 入队时按 `expire_ts` 最小堆排序;emergency 的 `expire_ts = UINT32_MAX` |
| B4 | RT-MQTT (Bhattacharya et al., 2019 改) | 文献基线 | 复现论文 §IV 算法:优先级分桶 + 超时提权,无 min_display,无 LRU |
| **B5** | **PA-MQS(本文)** | 优先级 + 老化 + min_display + 显式抢占规则 | 见 02-客户端固件规格.md §4.3 |

> **实现一致性保证**:6 个调度变体通过编译宏 `CONFIG_SCHEDULER_ALGO=0..5` 切换,所有其他代码路径(MQTT/TLS/LVGL/传感器)完全相同,排除非调度因素干扰。

### 2.3 工作负载(workload profile)
按 ITU-T G.1010 风格定义 3 个负载强度 × 4 种到达分布 = 12 个 cell,每 cell 跑 30min:

| 强度 | 平均到达率 λ | EMERG 占比 | WARN 占比 | NOTICE 占比 | INFO 占比 |
|---|---|---|---|---|---|
| Light  | 0.2 msg/s | 1% | 5%  | 30% | 64% |
| Medium | 0.5 msg/s | 2% | 10% | 35% | 53% |
| Heavy  | 1.0 msg/s | 5% | 15% | 40% | 40% |

到达分布参数:
- **Poisson**:`numpy.random.exponential(1/λ)` 作为消息间隔
- **Bursty**:Pareto burst(`shape=1.5, scale=0.1`),单次 burst 注入 3-8 条同级消息,间隔服从 Exp(0.3)
- **Diurnal**:λ 按 `0.2 + 0.8 × sin²(2π·t/T)` 日周期调制,T=30min 压缩到实验时间轴
- **Replay**:从 MSG-TRACE-4w 取同等时长片段,等比缩放时间戳到目标 λ

### 2.4 指标定义与计算

所有指标从 probe CSV 后处理得出,脚本 `experiments/pa_mqs/compute_metrics.py`:

```python
def e2e_latency_ms(probe_df, msg_id, pg):
    server_ts = pg.fetchval("SELECT inject_ts FROM msgs WHERE id=$1", msg_id)
    display_row = probe_df[(probe_df.event=='DISPLAY') & (probe_df.msg_id==msg_id)]
    if display_row.empty: return None  # 未上屏(饥饿/过期)
    return display_row.iloc[0]['wall_ts_ms'] - server_ts * 1000

def starvation_rate(probe_df):
    # INFO 消息在 TTL 到期前从未出现 DISPLAY 事件
    all_info = set(probe_df[probe_df.level=='info']['msg_id'])
    displayed = set(probe_df[(probe_df.event=='DISPLAY') &
                             (probe_df.level=='info')]['msg_id'])
    return len(all_info - displayed) / max(len(all_info), 1)

def user_miss_rate(probe_df):
    # DISPLAY 后不到 500ms 出现 PREEMPT = "用户一眼未看完"
    disp = probe_df[probe_df.event=='DISPLAY'].set_index('msg_id')['boot_ms']
    prmp = probe_df[probe_df.event=='PREEMPT'].set_index('msg_id')['boot_ms']
    common = disp.index.intersection(prmp.index)
    return (prmp[common] - disp[common] < 500).sum() / max(len(common), 1)
```

单台设备多次 run 的数据用箱线图呈现;跨 run 的 5/95 分位作为误差带。

| 指标 | 单位 | 重要性 |
|---|---|---|
| `emerg_p95` | ms | **主指标**,对应 H1 |
| `starvation_rate` | % | **主指标**,对应 H2 |
| `user_miss_rate` | % | 对应 H3 |
| `expire_rate` | % | 辅助 |
| `preempt_count` | 次/分钟 | 辅助 |
| `dedup_hit_rate` | % | LRU 有效性 |
| `msg_task_cpu` | % | `vTaskGetRunTimeStats` |
| `dram_peak` | KB | `heap_caps_get_minimum_free_size` 反推 |

### 2.6 故障注入(Fault Injection)

| 场景 | 注入脚本 | 触发时刻 | 通过标准 |
|---|---|---|---|
| Wi-Fi 闪断 30s | `fi_wifi_drop.sh`:`iw dev wlan0 disconnect` → 30s sleep → 重连 | 运行 15min 时 | 重连后 broker `(msg_id, device_id, ack_type)` 无重复条目;STARVE 事件数 = 0 |
| MQTT broker 抖动 60s | `fi_mqtt_drop.sh`:`iptables -I OUTPUT -p tcp --dport 8883 -j DROP` → 60s → `-D` | 运行 10min 时 | NVS `pending_ack.count` 最终归零;broker acknowledged 条数 = 端侧 ACK_SENT 条数 |
| 设备 brown-out 重启 | `fi_reboot.sh`:向端侧 UART 发 `reboot` 指令触发软复位 | 运行 20min 时 | 重启后 telemetry seq 从 NVS 接续不重置;服务端 `UNIQUE(device_id,seq)` 无冲突 |
| 时钟回拨 1h | `fi_sntp_spoof.py`:伪造 SNTP 应答注入 `now - 3600` | 启动前 | probe CSV 中 `boot_ms` 严格单调递增,无负值间隔(boot_ms fallback 生效) |

### 2.7 统计方法

**实验规模**:12 cell × 6 baseline × 5 rep × 1 device = 360 条样本序列(`make run-pa-mqs` 全自动,约 1 天)

**主对比步骤**:
1. 对每个 cell,计算 6 个 baseline 的 `emerg_p95` 中位数,得 72 个数据点
2. Friedman 检验(非参数,不假设正态):$H_0$ ="6 条曲线来自同分布"
3. 若 $p < 0.01$ 拒绝 $H_0$,用 Nemenyi 事后检验两两比较,画 Critical Difference 图
4. 效果量:Cohen's $d$ = (B5 − B0) / pooled_std;需 $d > 0.8$(large effect)才算有说服力
5. Bootstrap 1000 次 resample 报各 baseline 的 95% CI

- 图 6-2:Heavy+Replay 下 6 条 e2e latency CDF 曲线

### 2.8 与文献的差异化(写在贡献小节里)
PA-MQS ≠ "经典优先级 + 老化",其**新颖性**在于把"**当前显示项不可被同级抢占**" 这一**HCI 约束**形式化进了调度策略:
$$\text{preempt}(t, q_i) = (q_i.\text{level} > t.\text{level}) \land (\text{now} - t.\text{shown\_at} \ge \tau_{\min})$$
其中 $\tau_{\min} = 2000\text{ms}$。已有调度文献几乎都假设可任意抢占,本工作显式建模"必须保证人眼可读时长"。

## 3. HSDL-MCP 实验(对应第 8 章 "LLM 工具层")

### 3.1 假设
- **H4**:层级化工具(粗工具 → 细工具,Guard 二次确认)在 REDTEAM-300 上 **误调用率 ≤ 5%**,而扁平 function-calling 基线 ≥ 30%
- **H5**:在 ADMIN-100 真实任务上,HSDL-MCP 的**任务完成率不显著低于** 扁平基线(差值 < 5pp,p > 0.05)
- **H6**:平均**轮次/token 成本** 仅上升 ≤ 20%(可接受工程开销)

### 3.2 基线
| 编号 | 基线 | 说明 |
|---|---|---|
| C0 | OpenAI function-calling 扁平工具集 | 同样 8 个工具,无层级、无 Guard |
| C1 | 原生 MCP(无 sensitive flag) | 工具 schema 一致,但无二次确认 |
| C2 | MCP + 全工具都强制确认 | 极端对照,看用户体验代价 |

### 3.4 ADMIN-100 任务集与用户研究协议

**任务分布**:

{ "id": "A-067",
  "input": "给电气组广播:下午3点开会,需要确认",

**双盲协议**:8 名被试(6 研究生 + 2 企业 IT),拉丁方均衡 task 顺序,不告知背后实现。每人在 5 个系统上各完成 20 条(从 ADMIN-100 随机抽,不重叠),共 800 条任务记录。

**主观量表**:每名被试完成每个系统后立即填 SUS(10 题)+ NASA-TLX(6 维),总计 5 × 8 = 40 份量表。

### 3.5 指标与统计方法
| 指标 | 定义 | 计算方式 |
|---|---|---|
| `unauth_call_rate` | 红队 300 中绕过权限/二次确认的占比 | `redteam_runner.py` 解析 audit_log |
| `task_success_rate` | ADMIN-100 中端到端语义正确的占比 | 人工标注 |
| `avg_turns` | 每条任务平均 LLM↔工具 轮次 | SSE `tool_call` 事件计数 |
| `avg_tokens` | 每条任务平均 prompt+completion token | LLM API `usage` 字段 |
| `confirm_trigger_rate` | sensitive 调用中触发 need_confirm 的占比(目标 = 100%) | Guard 日志 |
| `false_confirm_rate` | 非敏感工具被误触发确认的占比(目标 < 5%) | Guard 日志 |
| `task_time_s` | 被试完成单条任务的墙钟时间 | 实验记录 |
| `SUS / NASA-TLX` | 标准量表分 | 问卷 |

**统计检验**:
- `task_success_rate` 5 系统对比:Friedman + Nemenyi
- C4 vs C0 的 `unauth_call_rate` 差值:Fisher 精确检验(两比例)
- SUS / TLX:Kruskal-Wallis + 两两 Mann-Whitney U(Bonferroni 校正)
- 效果量:Cliff's delta(非参数,适合有序数据)

### 3.6 消融
- **-Hierarchy**:工具扁平化(合并 `query_user` + `send_message` 为 `send_by_name`),预期 LLM 在模糊查询时误调用 broadcast 的频率上升
- **-Guard**:去二次确认(`sensitive=False`),预期 REDTEAM `unauth_call_rate` 与 C1 基线持平,直接证明 Guard 是保护层而非 schema 约束
- **-ArgsHashBinding**:token 仅绑定 `(op, exp)`,不绑参数哈希;手工构造 args_swap 攻击包,验证无哈希绑定时确实可被利用

### 3.7 形式化贡献
权限格 $L = (\text{Role}, \text{Operation}, \text{Scope})$,工具调用 $t = \langle \text{op}, \text{args}\rangle$ 仅当 $\langle \text{role}(u), t.\text{op}, \text{scope}(t.\text{args}) \rangle \in \text{Permit}$ 才执行。
**定理(论文里给证明草稿)**:在 Guard 满足两条不变量(单调审计 + token 一次性)前提下,任何能在被审计的会话中完成的敏感操作,**必有人类显式确认事件**与之一一对应。 → 即"零 LLM-自主执行敏感操作"形式可证。

## 4. EC-BAID 实验(对应第 9 章 "行为感知干预")

### 4.1 假设
- **H7**:加入个性化 7 日均值基线后,告警的 **F1** 比固定阈值基线高 ≥ 0.15(p < 0.01)
- **H8**:虚报率(False Positive Rate)下降 ≥ 50%
- **H9**:用户主观接受度(Likert-5)中位数 ≥ 4(扰民感低)

### 4.2 任务设定与 Ground Truth 标注

**预测单元**:5 分钟滑动窗口,步长 1 分钟,`should_intervene ∈ {0,1}`。

**三类正例**(分开报 per-class 指标):
| 类别 | 触发条件 | 占正例比 |
|---|---|---|
| Abnormal Static | `static_in_window > avg_static_7d × 1.5 AND static_in_window > 900s` | ~60% |
| Unack Emergency | emerg_displayed 后 5min 无 ack | ~30% |
| Fall Detection | `max_acc > 3g AND orientation_flip` | ~10% |

**双盲标注流程**:
```
1. 两名工程师独立对 BHV-LOG-4w 全量打标 (0/1)
2. 计算 Cohen κ;若 κ < 0.85 召开协商会,逐条讨论分歧
3. κ ≥ 0.85 接受;剩余分歧由第三位研究者最终裁决
4. 写入 bhv_labels.csv: device_id, window_start, label, category
   预期正例率 3-5%(告警事件自然稀疏)
```

### 4.3 特征工程(供 ML 基线与本方案共用)

每个 5min 窗口提取特征向量 $\mathbf{x} \in \mathbb{R}^{18}$:

```python
def extract_features(window_df, profile_7d):
    return {
        # === 窗口绝对指标 (9维) ===
        "static_secs":          window_df['static_secs'].sum(),
        "max_intensity":        window_df['intensity'].max(),
        "mean_intensity":       window_df['intensity'].mean(),
        "step_count":           window_df['step_inc'].sum(),
        "shake_count":          window_df['shake_inc'].sum(),
        "battery_mean":         window_df['battery'].mean(),
        "has_emerg_unacked":    int(has_unacked_emerg(window_df)),
        "max_acc_g":            window_df['max_acc'].max(),   # IMU 6轴峰值
        "orientation_flipped":  int(orientation_change(window_df) > 90),

        # === 个性化相对特征 (9维) ===
        "static_ratio":         safe_div(window_df['static_secs'].sum(),
                                         profile_7d['avg_static_7d'] * 300 + 1e-6),
        "intensity_ratio":      safe_div(window_df['intensity'].mean(),
                                         profile_7d['avg_intensity_7d'] + 1e-6),
        "steps_ratio":          safe_div(window_df['step_inc'].sum(),
                                         profile_7d['avg_steps_5min_7d'] + 1e-6),
        "response_rate_7d":     profile_7d['response_rate_7d'],
        "delivery_rate_7d":     profile_7d['delivery_rate_7d'],
        "hour_of_day":          window_df['time'].iloc[0].hour / 23.0,
        "day_of_week":          window_df['time'].iloc[0].dayofweek / 6.0,
        "time_since_last_ack":  safe_div(time_since_last_ack(window_df), 3600),
        "consec_static_windows": count_consecutive_static(window_df),
    }
```

D4(EC-BAID)将特征序列化为人话注入 LLM prompt:
```
设备 {device_id} 近5分钟:
  静止 {static_secs}s(个人7日均值{avg_static_7d}s,当前是均值的 {static_ratio:.1f} 倍)
  步数 {step_count}步(7日同时段均 {avg_steps_5min_7d}步)
  摇一摇次数 {shake_count},7日响应率 {response_rate_7d:.0%}
  有未确认紧急消息:{has_emerg_unacked}
请判断是否需要干预?JSON: {"intervene": true/false, "reason": "..."}
```

### 4.4 基线实现要点
| 编号 | 关键实现 |
|---|---|
| D0 | `static_secs > 1800 OR has_emerg_unacked OR max_acc_g > 3` 硬规则 |
| D1 | `static_secs > avg_static_7d×1.5 OR ...`;均值从 Redis profile 取 |
| D2 | LSTM:18维×12步(最近1h) → 隐层128 → sigmoid;Adam lr=1e-3;early_stop patience=5 |
| D3 | XGBoost:`n_estimators=200, max_depth=6, lr=0.05, scale_pos_weight=class_ratio`;SHAP 分析特征贡献 |
| D4 | 端侧 D0 粗筛(避免打每个窗口)→ 命中后调 LLM + 注入 profile_7d;解析 JSON `intervene` |

### 4.7 用户研究

**时间线**:
- 第 1-2 周(Baseline):固定阈值 D0 运行,5 名被试轮流佩戴,每天末填 5 题 Likert 问卷
- 第 3-4 周(Treatment):切换到 EC-BAID D4,同样问卷
- 第 4 周末:补 NPS + 开放问题

**问卷**:
```
Q1: "今天的提醒消息让我感到被打扰"  (反向,越低越好)
Q2: "今天的提醒帮助我及时注意到事情" (正向)
Q3: "提醒消息的时机感觉合适"         (正向)
Q4: "我愿意继续使用这个提醒功能"     (正向)
Q5: "提醒频率让我感到合适"           (正向)
```

**分析**:5 名被试 × D0/D4 两阶段配对 → Wilcoxon 符号秩检验(n=5 时只能报效果量与描述统计,p 值仅供参考);重点报 Q1 下降量(虚报感)和 Q2 上升量(有用性)。

### 4.8 消融
| 变体 | 去掉:个性化基线 | 去掉:LLM 决策 | 去掉:端侧粗筛 | 预期影响 |
|---|---|---|---|---|
| Full |   |   |   | F1 最高,成本合理 |
| -Personal | ✗ |   |   | FPR ↑(对"天生爱静"用户误报) |
| -LLM |   | ✗ |   | F1 ≈ D3(XGBoost 数值持平),但用户研究 Q2 显著下降(无解释文字) |
| -EdgeFilter |   |   | ✗ | LLM 调用 ×50,成本不可接受;F1 无变化(证明粗筛不影响精度只省钱) |

> **-LLM 变体**故意设计成"F1 持平":它证明 LLM 的价值**不在分类准确率**(XGBoost 够用),而在给干预消息附上**一句人话解释**("因为你已静止 35 分钟,比你平时多一倍"),这才是用户接受度提升的来源 — 由用户研究 Q2 数值佐证。

| 维度 | 测量项 | 工具 | 目标值(写进结论) |
|---|---|---|---|
| **能耗** | 电池续航(典型负载 100 msg/天 + 每日 2h 主动唤醒) | INA226 + 电流采集卡 | ≥ 7 天 |

## 6. 研究贡献(写到学位论文"绪论 §1.4 主要贡献"和"结论 §10")

### 6.1 学术贡献(三条,对齐三创新点)
1. **PA-MQS**:在嵌入式可穿戴设备的"人眼可读时长"硬约束下,提出兼顾**优先级、老化、抢占保护、去重**的统一调度算法;给出形式化抢占条件、复杂度分析($O(\log n)$ 入队 + $O(1)$ 调度决策)与可证伪实验,显著优于 5 条经典/文献基线。

3. **EC-BAID**:提出"**端侧粗筛 + 云侧 LLM + 个性化 7 日基线**"的端云协同行为干预架构,在 4 周真实部署的 BHV-LOG-4w 上 F1 提升 ≥ 0.15,虚报率下降 ≥ 50%,且 LLM 月度调用成本可控(< 30 次/设备/天)。
