## 一、项目综述

### 1.1 一句话定义 BeaconOps
基于 ESP32-C3 自研 36×19mm 可穿戴终端,以 MQTT + MCP 为通信骨架,通过服务端 LLM Agent 实现自然语言驱动的多设备调度、屏幕通知与流式 TTS 语音播报,适用于无手机/受控通信/保密场景的企业级智能调度系统。

### 1.2 核心价值主张
| 维度 | 价值 |
|---|---|
| **场景独占性** | 制造业车间、医院手术区、监狱、矿井、保密单位等**禁手机**场景 |
| **交互革命** | 管理员"说一句话",LLM 自动分解为多设备并发指令 |
| **可靠性** | 多级优先级 + 多层确认,emergency 消息 0 丢失 |
| **隐私可控** | 全栈私有部署,数据不出局域网 |
| **成本** | 单终端 BOM < 100 元(批量),远低于工业对讲机 |
| **交互形态** | 服务端→设备**主路径单向**,设备仅通过"摇一摇"回执确认收到 |

### 1.3 关键技术指标(目标)
- 端到端命令延迟 < 200ms(P95)
- 单服务节点支持 ≥ 1000 台并发设备
- 端侧续航 ≥ 1 天(典型负载,屏幕常亮 + Wi-Fi 常连)
- emergency 消息送达率 = 100%
- 控制台 LLM 复杂指令执行成功率 ≥ 90%
- 摇一摇确认识别率 ≥ 95%(创新点 3 子目标)

## 二、研究背景与问题定义

### 2.1 真实痛点
现代企业通信高度依赖智能手机 + IM(微信/钉钉/飞书),但以下场景**无法或不允许使用手机**:

| 场景 | 限制原因 |
|---|---|
| 精密制造车间 | 静电/油污/防爆 |
| 医院手术与 ICU | 卫生与电磁干扰 |
| 军工/科研保密单位 | 信息安全 |
| 监狱/法院羁押区 | 监管要求 |
| 矿井/隧道 | 信号缺失,需私有网络 |

现有替代方案及缺陷:

| 方案 | 缺陷 |
|---|---|
| 工业对讲机 | 全员广播、无定向、无回执、无屏幕、无智能调度 |
| 工业寻呼机 | 单向、显示能力弱、无 LLM 交互 |
| 工业平板 | 体积大、不便携、成本高 |
| 自建 IM | 仍需手机,核心问题未解 |

### 2.2 形式化的研究问题

> **问题 P**:在受控通信环境下,如何设计一种**轻量、可穿戴、可被自然语言指挥**的智能调度终端系统,使得管理者能以接近"自然对话"的方式精准调度物理空间内的人员?通信主路径为服务端→设备**单向下行**,设备侧通过 IMU"摇一摇"作为唯一回执通道,以最小化端侧主动上行的能耗与复杂度。

子问题:
- **P1**:在端侧资源极度受限(C3 仅 400KB SRAM、单核 RISC-V)条件下,如何保证多优先级消息的可靠投递与公平调度?
- **P2**:如何让 LLM 安全、低成本、可扩展地驱动设备群,而不是用穷举工具调用的方式?
- **P3**:如何在带宽与隐私约束下,实现端云协同的人员状态感知与主动干预?

→ 这三个子问题对应论文的三个创新点。


| 项 | 规格 |
|---|---|
| 尺寸 | 36.36 × 19.39 mm |
| 主控 | ESP32-C3 (RISC-V 单核 160MHz, 400KB SRAM, 4MB Flash) |
| 屏幕 | 1.47" IPS 172×320,SPI 接口 |
| 音频 | I²S DAC + 微型扬声器 |
| 传感器 | 6 轴 IMU(加速度+陀螺仪) |
| 电量计 | CW2017(I²C,SOC 精度 ±1%) |
| 电源 | DCDC 降压 + 线性充电 + USB/电池切换 |
| 存储 | 外置 SPI Flash |
| 通信 | Wi-Fi 2.4G(BLE 开启但不使用,节功耗)+ IPEX 外置天线 |
| 接口 | USB Type-C, 2pin 锂电接口, 触点排针 |

### 4.2 资源约束分析(论文中要明确写)
ESP32-C3 资源紧约束,**这正是创新点 1 的研究动机**:

| 资源 | 容量 | 已知占用 | 可用 |
|---|---|---|---|
| SRAM | 400 KB | OS + Wi-Fi 协议栈 ~250KB | **~150 KB 给应用** |
| Flash | 4 MB | bootloader + partition + LVGL + libopus + 字库 + TTS缓存 紧凑 | 空间不够双分区 OTA,固件走 USB 烧录维护 |
| CPU | 160 MHz 单核 | LVGL 渲染 + MQTT + 音频解码并发 | **必须做严格任务调度** |
| AI 加速 | **无** | — | **不能跑 ML 推理(否定 ML 行为识别)** |

→ **设计原则**:任何方案都必须在 150KB SRAM、单核 CPU 下可行,这是论文的"约束条件"。

## 五、消息描述语言与协议设计

### 5.1 设计原则
1. **显示与音频解耦**:title/text/audio 三者独立可选
2. **优先级驱动 UI 与行为**:level 字段决定主题色/振动/确认方式
3. **TTL + 去重 + ack**:适配不可靠网络
4. **可扩展但紧凑**:JSON 起步,后续可演进为 CBOR

### 5.2 消息 Schema(JSON)
```json

  "ts": 1715587200,
  "ttl": 3600,

  "source": {
    "type": "human | llm | system",
    "operator": "admin_zhao"
  },
  "display": {
    "title": "开会通知",
    "text": "下午3点到3楼会议室",
```

### 5.3 MQTT Topic 命名空间
```
device/{device_id}/cmd          # ⬇ 服务器 → 设备:命令
device/{device_id}/ack          # ⬆ 设备 → 服务器:回执(delivered/displayed/acknowledged)
device/{device_id}/telemetry    # ⬆ 心跳 + 行为状态 + 电量
device/{device_id}/status       # ⬆ LWT 在线状态
device/{device_id}/audio        # ⬇ 流式音频分块
broadcast/role/{role}/cmd       # ⬇ 按角色广播
broadcast/dept/{dept}/cmd       # ⬇ 按部门广播
broadcast/all/cmd               # ⬇ 全员广播
system/time                     # ⬇ 时间同步
```

### 5.4 多层确认机制
| 层级 | 触发 | 用途 |
|---|---|---|
| `delivered` | MQTT QoS 1 PUBACK | 网络层送达 |
| `displayed` | 屏幕真正绘制 | 应用层送达 |
| `acknowledged` | 用户敲击/按键确认 | 用户层送达(emergency 必需) |
| `escalated` | 超时未 ack | 自动升级(转电话/上级通知) |

## 六、端侧固件设计

### 6.1 任务划分(FreeRTOS)
| 任务 | 优先级 | 栈 | 职责 |
|---|---|---|---|
| `net_task` | 5 | 8KB | MQTT 连接维护、断线重连 |
| `msg_task` | 6 | 4KB | PA-MQS 队列调度、消息生命周期 |
| `ui_task` | 4 | 12KB | LVGL 渲染、屏幕状态机 |
| `audio_task` | 3 | 8KB | I²S 流式 Opus 解码与播放 |
| `sensor_task` | 2 | 4KB | IMU 采样 + 行为状态机 |
| `power_task` | 1 | 2KB | 电量监控 + 睡眠调度 |

### 6.2 LVGL 极致裁剪清单
```c
#define LV_MEM_SIZE             (32 * 1024)  // 32KB 堆
#define LV_COLOR_DEPTH          16           // RGB565
#define LV_DRAW_SW_COMPLEX      0            // 关渐变/阴影/模糊
#define LV_USE_ANIMATION        1            // 仅淡入淡出
#define LV_USE_CHART            0
#define LV_USE_TABLE            0
#define LV_USE_KEYBOARD         0
#define LV_USE_CALENDAR         0
// 字库:lv_font_conv 子集化,只打包常用 3500 中文字
// 16px 中文字库 ≈ 200KB Flash
```

### 6.3 屏幕状态机(手动确认 / FIFO 严格依序)
```
        ┌──────┐
        │ Home │ ←──────────────────┐
        └──┬───┘                     │
           │ 队列非空                │ 队列空
           ▼                         │
       ┌────────┐  摇一摇 ack        │
       │ Show 1 ├──────────────┐     │
       └────────┘              ▼     │
           │             ┌──────────┐│
           │ EMERG 抢占  │  Next/   ├┘
           ▼             │  Home    │
       ┌────────┐  ack   └──────────┘
       │ EMERG  ├──→ 原 Show 1 压回队列首
       └────────┘     再来 EMERG 继续压
                      → 调度器仍按优先级 + FIFO 选下一条

   任意状态(背光独立计时):
     10s 静止 → 背光关 (缓动 0%)
     IMU 事件 → 背光 100% (cubic ease,1000ms wake_grace 内 shake 不作 ack)
```

**原则**:
- 默认不倒计时,不自动切换,只靠摇一摇
- EMERG 不去重 / 不合并,新 EMERG 直接置顶,旧项压回队列 FIFO 按优先级入位
- 可选:服务端下发消息可设 `display.auto_dismiss_ms` 覆盖默认(谁调谁负责)

### 6.4 流式音频管线
```
MQTT 分块到达 → ringbuffer (4KB) → opus_decoder → I²S DMA 双缓冲 → 喇叭
                                ↑
                          边收边解,首包延迟 < 300ms
```

### 6.5 行为识别(端侧规则法)
1秒滑动窗口 + 加速度模长方差,详见 [§10.3](#103-创新点-3ec-baid)。

## 八、LLM + MCP 调度中枢

### 8.1 为什么用 MCP 而不是 Function Calling
| 维度 | 原生 Function Calling | MCP |
|---|---|---|
| 工具发现 | 静态注入 prompt | 动态发现 + 热加载 |
| 跨进程 | 难 | 标准协议(stdio/sse) |
| 权限边界 | 全靠 prompt 约束(易绕过) | **服务端拦截** |
| 多 LLM 共享 | 重复实现 | 一次实现多端共用 |
| 工具版本管理 | 无 | 有 schema 版本 |

### 8.2 MCP 工具集核心 Schema(对应创新点 2)
```python
@mcp.tool()
def query_user(query: str, role: str = None,
               dept: str = None) -> list[dict]:
    """模糊查询用户。返回 user_id, name, role, dept, online_status。"""

@mcp.tool()
def send_message(target_user_id: str,
                 title: str = None, text: str = None,
                 voice_text: str = None,
                 level: Literal["info","notice","warn","emergency"] = "info",
                 schedule_at: int = None,
                 require_ack: bool = False) -> dict:
    """向指定用户发送消息。schedule_at 为 unix 秒,留空立即发送。"""

@mcp.tool(sensitive=True)        # ← 二次确认
def broadcast(scope_type: Literal["role","dept","all"],
              scope_value: str,
              title: str = None, text: str = None,
              voice_text: str = None,
              level: str = "notice") -> dict:
    """向角色/部门/全员广播。scope=all 或 level=emergency 强制人类二次确认。"""

@mcp.tool()
def get_user_status(user_id: str) -> dict:
    """获取用户实时状态:online/battery/current_state/last_active。"""

@mcp.tool()
def get_device_profile(device_id: str) -> dict:
    """返回画像:今日 static_total/steps/shake_count/manual_ack/auto_ack +
       7d 滑动均值 avg_static_7d/response_rate_7d/delivery_rate_7d + last_active_ts。
       前台对话也可调用,用于 LLM 个性化决策。"""

@mcp.tool()
def list_pending(target: str = None) -> list[dict]:
    """查询待 ack 消息。"""

@mcp.tool()
def cancel_scheduled(task_id: str) -> dict:
    """取消未触发的定时任务。"""

@mcp.tool(sensitive=True)        # ← 二次确认
def delete_device(device_id: str) -> dict:
    """注销/解绑设备(毁灭性)。"""
```

### 8.3 权限拦截 + 敏感操作二次确认(创新点 2 的关键)
所有工具调用 → MCP Server Guard → 检查调用者 RBAC → 按需重写 args 或拒绝。
```python
# 形式化:权限格 L = (Role, Operation, Scope)
# 工具调用 t 仅当 t ∈ permit(L) 时执行,否则返回 PermissionDeniedError
```

**敏感工具(`sensitive=True` 或参数命中危险范围:scope=all、level=emergency、delete_*)**:
1. LLM 调用 → Guard **不直接执行**,返回 `{status:"need_confirm", confirm_token, plan}`
2. 前端 `ToolCallCard` 渲染 plan + [确认执行] / [取消] 按钮
3. 用户点确认 → `POST /llm/confirm {token}` → Guard 校验 token 后真正执行
4. 60s 未确认 / 取消 → token 作废
5. `audit_log` 双阶段记录(propose / confirm),操作主体 = 当前登录人(非 LLM)

> **目的**:即使 LLM 被 prompt injection 诱导调用 `broadcast(scope=all)` 或 `delete_device`,
> 也必须经过人类一键确认,屏蔽"模型一时糊涂引发企业级事故"的风险面。
> token 由服务端签名,绑定 `(operator_id, tool_name, args_hash, exp)`,防重放。

## 十、三大学术创新点

### 10.1 创新点 1:PA-MQS

**全称**:Priority-Aware Message Queue Scheduling for Constrained IoT Devices

**问题建模**:
- 队列容量 C(端侧固定 16)
- 消息流 M = {m₁, m₂, ...},每条 mᵢ = (level, ttl, payload_size, arrive_at)
- 优先级权重 w = {info:1, notice:2, warn:5, emergency:100}
- 目标:在 SRAM 受限下,**最大化加权送达率**且 **emergency 必达**

**算法核心**:
```
1. 准入控制
   if level == emergency: force_admit, evict_lowest_priority_if_needed
   else if queue_full: compare urgency_score, drop minimum
2. 抢占规则
   level_new > level_current  →  抢占当前显示
3. 老化(防饥饿)
   每 Δt(=10s),队列内消息 effective_priority += aging_step
4. 反馈调整(在线学习)
   若 ack 延迟 > SLA,提升对应 level 的 w
```

**复杂度**:O(log C) 入队(堆),O(1) 出队。

**实验对比基线**:FIFO / SP / WRR / PA-MQS

**评价指标**:
- emergency 送达率(目标 100%)
- 平均加权延迟
- info 饥饿率
- SRAM 峰值占用

**实验装置**:Python paho-mqtt 模拟器,泊松分布 + 优先级混合,5 种负载场景 × 4 算法 × 1 万条消息。

---

### 10.2 创新点 2:HSDL-MCP

**全称**:Hierarchical Semantic Device Layer over MCP

**问题建模**:
- 设备数 N(目标 1000+)
- 朴素方案:为每个设备暴露 K 个工具 → 工具数 = N×K → token 消耗 O(N)
- 本文方案:暴露 O(1) 个语义工具 + O(log N) 索引查询

**三大机制**:
1. **设备空间语义压缩**:用 `query_user(自然语言查询)` 替代逐设备工具
2. **权限感知拦截**:权限格 L = (Role, Op, Scope),Guard 在工具调用前裁剪
3. **意图分解(DAG)**:复杂指令 → 并行/串行子调用图

**实验对比基线**:
- 原生 OpenAI Function Calling(穷举工具)
- LangChain Agent(ReAct)
- 自实现 MCP(无 HSDL)
- **HSDL-MCP**

**评价指标**:
- 复杂指令成功率(测试集 100 条)
- 平均 token 消耗(prompt + completion)
- 端到端调度延迟
- 权限违规拦截率(注入测试 50 条)
- 工具空间膨胀曲线(N = 10/100/1000)

**实验装置**:GPT-4o-mini 与本地 Qwen2.5-14B 双模型,自标注调度指令测试集。

---

### 10.3 创新点 3:EC-BAID

**全称**:Edge-Cloud Behavior-Aware Intervention Decision

**问题建模**:
- 端侧采样率 50Hz IMU,带宽预算 B = 1KB/min/设备
- 云侧需要做的不是"分类",而是"**是否干预 + 何种干预**"
- 目标:在带宽 B 内,最大化干预 Precision/Recall

**两层架构**:
1. **端侧(规则法)**:
   ```
   energy = Σ(|a| - g)² / N (1s 窗口)
   state = STATIC | WALK_SLOW | WALK_FAST | RUN | SHAKE
   intensity = clamp(round(2·log10(E+1)·5), 1, 10)
   step ← 峰值检测 + 250ms 去抖
   静止时长 = state==STATIC 累计
   ```
   上报频率:状态变化时 + 5min 心跳

2. **云侧(LLM-as-Decider)**:
   - 输入:[state_seq(24h), role, recent_messages, time_of_day, battery]
   - prompt 模板内置干预策略
   - 输出:JSON {should_intervene, intervention_type, content, level}

**实验对比基线**:
- B0:纯规则(if static>30min then notify)
- B1:端侧 LSTM 分类 + 规则触发
- B2:全数据上云 + LLM
- **EC-BAID**(端侧规则 + 云侧 LLM)

**评价指标**:
- 干预 Precision / Recall / F1
- 上行带宽消耗(KB/h/设备)
- 端到端响应延迟
- 用户主观接受度(5 个被试 × 3 天问卷)

## 十一、安全设计

### 11.1 设备认证
- 出厂烧录唯一设备证书(X.509)
- MQTT TLS 双向认证
- 设备私钥存 eFuse,不可读出

### 11.2 命令防伪
- 每条命令包含 `cmd_id`(uuid)+ HMAC 签名(可选)
- 端侧维护已执行 ID 集合(LRU 1000 条),拒绝重放

### 11.3 控制台
- JWT + 短期 token + RefreshToken
- 高危操作(emergency 广播、删除设备)二次确认 + 审计日志
- LLM 调用强制经过 MCP Guard,**不允许直连数据库**

### 11.4 隐私
- 行为数据默认本地化部署
- 公司内 Top-1 ACL 控制行为数据可见范围
- 用户可随时下载/删除自己的数据(GDPR-compliant)
