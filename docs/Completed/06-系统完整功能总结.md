# BeaconOps 系统完整功能总结

> 文档目标：用一份文件把 BeaconOps 当前已经具备的完整功能讲清楚。
> 文档口径：只描述当前系统真实具备的能力，不展开实现代码，不引用旧版方案稿。

---

## 1. 系统是什么

BeaconOps 是一套面向 Beacon 终端的消息发布、设备管理、状态观测和行为回传系统。

它由四个部分组成：

1. 设备端硬件与固件。
2. 服务器端业务系统。
3. Web 控制台前端。
4. 设备与服务器之间的 MQTT 协议和数据模型。

系统的核心目标不是做通用聊天平台，也不是做复杂的 AI 助手系统，而是把一批 Beacon 终端组织起来，支持以下几件事：

1. 让设备稳定接入网络和服务器。
2. 让管理端向设备下发消息。
3. 让设备回传送达、显示、确认、过期等状态。
4. 让设备持续回传健康状态和行为状态。
5. 让前端控制台实时看到设备、消息和批次变化。

## 2. 系统由哪些对象构成

### 2.1 管理对象

| 对象 | 作用 |
|---|---|
| 管理员 | 登录控制台、发送消息、管理批次、设备和账号 |
| 批次 | 一组设备的授权边界、密钥边界和容量边界 |
| 设备 | 真实 Beacon 终端，按设备号唯一识别 |
| 消息 | 从控制台下发到设备的业务内容 |
| 审计记录 | 记录后台关键操作 |
| 行为记录 | 记录设备端按时间窗口汇总的活动数据 |

### 2.2 设备身份模型

系统使用两层设备身份：

1. 设备号。
2. 批次号。

设备号来自设备芯片自身的 MAC，运行时生成，天然唯一。

批次号来自固件编译配置，用于把一组设备归到同一个授权域下。服务器根据批次决定：

1. 这台设备是否允许接入。
2. 这台设备属于哪个批次。
3. 这台设备是否受到批次撤销影响。
4. 这台设备能否通过批次密钥完成鉴权。

## 3. 设备端完整功能

## 3.1 基础硬件能力

设备端当前已经实际接入并参与系统运行的硬件能力包括：

1. 显示屏，用于显示首页状态、消息卡片和提示信息。
2. 背光控制，用于唤醒和可见性控制。
3. IMU 传感器，用于检测运动、摇一摇和行为强度。
4. 电量计，用于读取电池电量。
5. 充电检测，用于判断充电状态。
6. 音频输出，用于本地提示音和确认音。
7. SPIFFS 存储，用于离线缓存部分业务数据。
8. NVS 持久化，用于保存设置和待发 ACK。
9. Wi-Fi 无线连接，用于接入网络。
10. SNTP 时间同步，用于建立可信时间基线。

这意味着设备不是静态展示终端，而是一个具备联网、显示、感知、缓存和回传能力的完整节点。

## 3.2 开机后的自动工作流程

设备上电后，当前会自动完成以下流程：

1. 初始化显示、传感器、音频、背光、电量计、存储等基础硬件。
2. 初始化 UI、提示音和本地状态模块。
3. 初始化 Wi-Fi 管理句柄。
4. 自动连接已配置的 Wi-Fi。
5. 连接成功后完成 DNS 设置和网络优化。
6. 同步网络时间。
7. 读取设备号并准备批次凭证。
8. 建立 MQTT 连接。
9. 启动消息、ACK、健康上报、行为聚合等业务模块。

整个过程不依赖现场人工逐步操作，属于自动拉起式设备。

## 3.3 设备接入服务器的方式

设备连接服务器时，不使用固定明文口令长期接入，而是使用动态签名口令。

当前接入模型如下：

1. 设备号作为 MQTT 的 client_id。
2. 批次号作为 MQTT 的 username。
3. 一次性签名口令作为 MQTT 的 password。
4. 签名口令由当前时间、随机数和批次密钥共同生成。

这个机制带来几个结果：

1. 设备不能随便伪造身份接入。
2. 旧口令不能长期复用。
3. 批次撤销后，整批设备都可以被服务端拒绝。
4. 同一随机数不能重复使用，服务端会做防重放检查。

## 3.4 设备消息接收能力

设备端当前支持接收三类下行目标：

1. 单设备消息。
2. 全员广播消息。
3. 部门广播消息。

当前系统的按部门发送已经成立，其实现方式是由服务器根据设备部门筛选目标设备，再按设备逐台下发消息，因此设备端不需要在运行时感知自己属于哪个部门。协议层虽然预留了部门广播的 topic 方案，但当前主链路并不依赖它。

设备收到消息后，会完成以下动作：

1. 判断消息格式是否合法。
2. 判断消息级别是否合法。
3. 判断显示内容是否为空。
4. 判断 ACK 要求是否合法。
5. 把通过校验的消息纳入本地队列。
6. 触发 UI 显示和提示音逻辑。

## 3.5 设备支持的消息级别

当前设备和服务器共同使用四种消息级别：

| 级别 | 作用 |
|---|---|
| info | 一般通知 |
| notice | 提醒类通知 |
| warn | 警告类消息 |
| emergency | 紧急类消息 |

其中 warn 和 emergency 在系统中有更强的确认要求，不是普通展示后就结束。

## 3.6 设备支持的消息确认语义

设备支持四类 ACK 节点：

| ACK 类型 | 含义 |
|---|---|
| received | 设备已经接收到并接纳该消息 |
| displayed | 该消息已经真正显示到屏幕上 |
| acknowledged | 用户已经通过摇一摇等交互完成确认 |
| expired | 非紧急消息超过有效期仍未确认 |

这些状态使后台不只是知道“消息发过了”，而是能区分：

1. 有没有收到。
2. 有没有显示。
3. 有没有被用户确认。
4. 是否已经超时失效。

## 3.7 设备本地消息表现能力

设备端当前不只是收包入库，还具备完整的本地交互能力：

1. 首页持续展示设备基本状态。
2. 消息到达后可以在 UI 中显示消息卡片。
3. 消息接纳时会触发提示音。
4. 用户确认后会触发确认音。
5. 背光会配合重要消息与交互事件增强可见性。
6. 当前显示项、提示信息和状态信息可以同时存在于同一设备 UI 体系中。

## 3.8 设备 ACK 可靠投递能力

设备并不是把 ACK 发一次就算结束，而是做了可靠投递处理。

当前 ACK 子系统具备以下能力：

1. ACK 会进入本地待发队列。
2. 待发队列会持久化到 NVS，不怕设备重启立即丢失。
3. ACK 支持定时重试。
4. 不同 ACK 类型使用不同重要性等级。
5. acknowledged 的可靠性等级最高。
6. 达到重试上限后会明确标记失败，而不是悄悄丢掉。

这意味着：即使设备临时断网，重要 ACK 也仍有较高概率在后续补发成功。

## 3.9 设备端异常事件上报能力

设备端当前能够主动上报几类异常事件：

1. ACK 重试到上限仍失败。
2. 下行消息解析失败。
3. 本地存储空间或缓冲空间不足。
4. 去重命中后主动丢弃重复消息。

这些事件让后台能区分“消息业务没成功”到底是因为：

1. 设备没收到。
2. 设备收到了但格式不合法。
3. 设备本地资源不足。
4. ACK 发送链本身出了问题。

## 3.10 设备健康上报能力

设备会按固定周期和状态变化两种方式回传健康数据。

当前健康上报包含以下维度：

1. 固件版本。
2. 批次标识。
3. 时间戳。
4. 电量百分比。
5. 是否充电。
6. Wi-Fi RSSI。
7. 当前 IP。
8. 运行时长。
9. SPIFFS 待发数量。
10. ACK 待发数量。
11. 丢弃计数。

健康上报不是只给服务器做日志记录，它直接驱动前端设备详情中的电量、RSSI、最后上报等信息。

## 3.11 设备行为识别与上报能力

设备端当前已经具备基于 IMU 的行为识别与时间窗口汇总能力。

当前行为分为五类：

| 行为 | 含义 |
|---|---|
| 静止 | 基本无活动 |
| 低幅活动 | 轻度移动 |
| 中幅活动 | 中等活动 |
| 高幅活动 | 较高强度活动 |
| 剧烈波动 | 强震动、跌倒样或强波动状态 |

设备不会把每一帧传感器原始数据直接上报，而是按时间窗口汇总为：

1. 每类行为持续了多少秒。
2. 发生了多少次摇晃。
3. 平均强度是多少。
4. 当日步数是多少。

这套机制让系统能用较低通信成本获得有业务意义的行为数据。

## 3.12 设备离线缓冲能力

当前设备端有两套离线缓冲机制：

1. ACK 离线缓冲。
2. profile 行为数据离线缓冲。

这意味着在网络短暂中断期间，系统仍然可以保住关键状态数据，待联网后继续补发，而不是完全丢失。

## 4. 服务器端完整功能

## 4.1 服务器总体职责

服务器端是整个系统的核心控制中枢，当前负责：

1. 管理员登录与权限控制。
2. 设备接入鉴权。
3. 批次管理。
4. 设备信息管理。
5. 消息创建、投递和重试。
6. ACK、事件、健康、行为数据接收与入库。
7. 实时事件推送给前端。
8. 审计记录。
9. 控制台静态资源承载。

## 4.2 管理员认证与权限功能

服务器端当前支持后台账号体系，具备以下功能：

1. 管理员登录。
2. 登录态保持。
3. 退出登录。
4. 基于角色的权限控制。
5. 基于 Cookie 的会话维护。
6. 基于 CSRF 的请求保护。

当前角色分为三类：

| 角色 | 能力 |
|---|---|
| admin | 拥有全部后台能力，包括管理员管理和审计查看 |
| operator | 可以发送消息、修改设备和批次，但不能管理管理员账号 |
| viewer | 只能查看，不能发送和修改 |

## 4.3 设备接入鉴权功能

服务器并不默认信任所有设备连接，而是执行完整的设备接入校验。

当前鉴权会检查：

1. 设备号格式是否正确。
2. 批次号是否存在。
3. 口令格式是否正确。
4. 时间偏差是否在允许范围内。
5. 批次是否已被撤销。
6. 设备是否已被禁用。
7. 该批次是否已经达到可接入容量。
8. HMAC 是否匹配。
9. 随机数是否重复使用。

鉴权成功后，服务器会：

1. 首次见到设备时自动登记设备。
2. 记录该设备属于哪个批次。
3. 更新最近见到时间。
4. 为该设备生成允许访问的 topic 权限。

## 4.4 批次管理功能

批次是系统中最关键的管理单元之一。当前服务器支持：

1. 创建批次。
2. 查看批次列表。
3. 查看批次详情。
4. 修改批次别名、备注和计划数量。
5. 撤销批次。
6. 恢复批次。
7. 删除已撤销批次。
8. 旋转批次密钥。

批次不仅是一个标签，它还承担以下功能：

1. 控制设备能否合法接入。
2. 限制设备数量上限。
3. 作为设备分组和归属依据。
4. 在被撤销时统一切断整批设备的授权。

## 4.5 设备管理功能

服务器端当前支持设备级管理，功能包括：

1. 查看设备列表。
2. 查看单设备详情。
3. 按批次筛选设备。
4. 按部门筛选设备。
5. 按在线状态筛选设备。
6. 按启用状态筛选设备。
7. 按关键字搜索设备号和别名。
8. 修改设备别名。
9. 修改设备部门。
10. 查看某台设备最近关联的消息。
11. 查看某台设备的行为时间轴数据。

设备详情不是静态名片，而是一个持续反映运行情况的观察页。

## 4.6 消息下发功能

服务器支持从后台创建并下发消息。当前消息支持三种目标：

1. 单设备。
2. 全员。
3. 部门。

创建消息时，当前可指定：

1. 目标类型。
2. 目标值。
3. 消息级别。
4. 标题。
5. 正文。
6. TTL。
7. ACK 要求。

服务器会在消息创建后：

1. 写入消息主表。
2. 计算所有 recipient。
3. 为每个 recipient 建立独立投递状态。
4. 立即尝试向在线设备下发。
5. 把离线设备保留在待投递状态。

## 4.7 消息重试与补发功能

服务器端消息系统不是“一发了之”，而是具备 recipient 级别的重试与补发能力。

当前具备以下特性：

1. 每个设备的消息状态单独跟踪。
2. 在线设备可立即尝试投递。
3. 离线设备上线后会自动补发。
4. 已发送但未完成 ACK 的消息会按退避策略重试。
5. 超过重试上限的 recipient 会被标记为失败。
6. 消息整体状态会根据所有 recipient 的状态重新计算。

因此，系统能准确回答以下问题：

1. 这条消息发给了哪些设备。
2. 哪些设备已经显示。
3. 哪些设备已经确认。
4. 哪些设备已经过期。
5. 哪些设备失败了。

## 4.8 设备上行接收功能

服务器端当前接收五类设备上行：

1. status。
2. ack。
3. event。
4. health。
5. profile。

每类上行的作用不同：

| 类型 | 作用 |
|---|---|
| status | 更新设备在线状态和固件版本 |
| ack | 推进消息状态 |
| event | 记录端侧异常或补充事件 |
| health | 更新设备当前健康快照 |
| profile | 记录设备行为窗口数据 |

## 4.9 行为数据聚合功能

服务器收到 profile 后，不只是原样保存，还支持对行为数据按时间范围和时间桶进行聚合查询。

这使服务器具备：

1. 保存原始行为窗口记录。
2. 按设备号查询行为数据。
3. 按 since、until 和 bucket 大小汇总数据。
4. 为前端图表生成补齐时间桶的连续时间轴。

这正是当前设备详情行为图的后端基础。

## 4.10 健康快照与可选历史功能

服务器会把 health 数据写入设备表中的当前快照字段，因此前端总能拿到设备最近状态。

此外，服务器保留了健康历史表和写入能力，但默认不开启长期写入。这意味着当前系统重点在“最新状态可观测”，而不是默认开启长期健康时序分析。

## 4.11 实时事件推送功能

服务器端当前使用 SSE 把关键变化实时推给前端。

实时推送覆盖以下范围：

1. 管理员登录和管理员变更。
2. 批次创建、修改、撤销、恢复、删除。
3. 设备上线、离线、信息更新、健康更新、行为更新。
4. 消息创建。
5. recipient 状态变化。
6. 消息事件变化。

因此，前端不需要不断全量轮询，就能在多数场景中及时感知变化。

## 4.12 审计功能

服务器端对关键后台操作保留审计记录。当前能够记录的范围包括：

1. 登录成功与失败。
2. 发送消息。
3. 创建、修改、删除管理员。
4. 创建、修改、撤销、恢复、删除批次。
5. 修改设备信息。

审计记录保留以下维度：

1. 时间。
2. 操作者类型。
3. 操作者名称。
4. 动作名称。
5. 目标类型。
6. 目标值。
7. 结构化详情。
8. 来源 IP。

## 5. 前端控制台完整功能

## 5.1 前端总体定位

前端是这套系统的统一操作台，当前主要承担三种角色：

1. 后台发送消息的操作入口。
2. 设备、批次和管理员的管理入口。
3. 设备状态、消息状态和行为数据的观察入口。

## 5.2 登录与权限感知功能

前端当前支持：

1. 登录页。
2. 登录态刷新。
3. 退出登录。
4. 角色差异化展示。
5. 页面内的按钮级权限限制。

因此同一套前端可以为 admin、operator、viewer 三种角色提供不同使用范围。

## 5.3 首页功能

当前首页的定位是简洁入口页，具备：

1. 系统标题展示。
2. 系统一句话介绍。
3. 消息到终端的可视化示意。
4. 到发送页的快捷入口。
5. 与实时连接状态联动的页面状态提示。

首页不是复杂总览大屏，而是轻量的系统入口。

## 5.4 发消息页功能

发消息页当前是完整可用的业务页，支持：

1. 选择目标类型为设备、部门或全员。
2. 在设备模式下搜索和多选设备。
3. 在部门模式下按已有部门值选择目标。
4. 查看当前选中目标的设备数量和在线数量。
5. 选择消息级别。
6. 填写标题和正文。
7. 发送成功后跳转到历史页或详情页。
8. 对 viewer 角色显示只读限制。

这意味着前端已经形成可直接操作的发送流程，而不是只有一个简单表单。

## 5.5 历史消息页功能

历史页支持：

## 6. 协议与数据模型完整功能

## 6.1 当前协议分工

系统当前采用 MQTT 作为设备和服务器之间的主要业务通道。

协议分工非常明确：

1. cmd 用于服务器下发消息。
2. ack 用于设备回执消息阶段状态。
3. event 用于设备上报异常事件。
4. health 用于设备上报健康快照。
5. profile 用于设备上报行为窗口。
6. status 用于设备在线离线状态。

这套拆分避免所有状态混在一种消息里，便于服务器按类型处理。

## 6.2 当前数据表层功能

服务器当前数据库承担以下持久化能力：

1. 保存批次与批次密钥。
2. 保存设备主档和最新健康状态。
3. 保存消息主记录。
4. 保存消息对每台设备的投递状态。
5. 保存消息事件。
6. 保存可选健康历史。
7. 保存管理员账号。
8. 保存审计日志。
9. 保存 MQTT 防重放随机数。
10. 保存行为时间窗口记录。

这意味着系统的主要业务对象都有明确落点，不依赖临时内存态维持运行。

## 6.3 当前消息状态模型

系统当前不是只区分“成功”和“失败”，而是使用细化状态来表达消息过程。

消息生命周期里最核心的状态包括：

1. queued。
2. sent。
3. delivered。
4. displayed。
5. acknowledged。
6. expired。
7. failed。

这使系统能从运营角度判断：

1. 消息有没有真正送到设备。
2. 消息有没有真正被看到。
3. 消息有没有被用户确认。
4. 失败发生在投递前、中还是后。

## 6.4 当前行为数据模型

行为数据不是一串模糊标签，而是按窗口保存结构化统计，包括：

1. 每种行为在该窗口中的时长。
2. 摇晃次数。
3. 平均强度。
4. 步数。
5. 记录时间。
6. 接收时间。

因此，前端图表展示的数据是有完整来源和完整结构的，不是推测性可视化。

## 7. 当前系统形成了哪些完整闭环

## 7.1 设备上线闭环

系统当前已经形成完整的设备上线闭环：

1. 设备上电。
2. 自动联网。
3. 时间同步。
4. 生成签名口令。
5. 服务器完成设备接入鉴权。
6. 设备进入批次管理范围。
7. 前端设备列表能看到这台设备。

## 7.2 消息下发闭环

系统当前已经形成完整的消息下发闭环：

1. 管理员在前端发起消息。
2. 服务器创建消息和 recipient。
3. 在线设备立即收到下行命令。
4. 设备进行本地展示和提示。
5. 设备回传 received、displayed、acknowledged 或 expired。
6. 服务器更新状态。
7. 前端历史页和详情页实时刷新。

## 7.3 离线补发闭环

系统当前已经形成离线补发闭环：

1. 设备不在线时，recipient 保持待发送。
2. 设备重新上线后，服务器自动重新扫描待发消息。
3. 设备收到历史待发消息。
4. 消息状态继续推进。

## 7.4 健康观测闭环

系统当前已经形成健康观测闭环：

1. 设备周期或按变化上报健康数据。
2. 服务器更新设备健康快照。
3. 服务器通过 SSE 推给前端。
4. 前端设备详情实时显示最新健康状态。

## 7.5 行为可视化闭环

系统当前已经形成行为可视化闭环：

1. IMU 持续采样。
2. 固件聚合行为窗口。
3. 设备上报 profile。
4. 服务器落库 behavior。
5. 前端请求行为时间轴接口。
6. 设备详情页绘制行为图。
7. 新的 behavior 到达时前端自动刷新。

## 7.6 批次撤销闭环

系统当前已经形成批次撤销闭环：

1. 后台撤销某个批次。
2. 服务器更新批次状态。
3. 该批次设备被统一置为不可用状态。
4. 服务器触发 broker 侧连接失效处理。
5. 这批设备后续无法再按原授权继续正常上行。
6. 前端批次页、设备页会反映该变化。

## 8. 这套系统现在能解决什么问题

从功能角度看，当前系统已经能解决以下实际问题：

1. 如何安全地把一批 Beacon 终端接入同一套后台。
2. 如何按设备、按部门、按全员三种方式发消息。
3. 如何区分消息是否真正被设备接收、显示和确认。
4. 如何在设备暂时离线时保留投递机会。
5. 如何观察设备当前是否在线、信号如何、电量如何。
6. 如何持续获得设备行为数据，而不是只知道设备是否在线。
7. 如何在前端实时看到消息、设备和批次变化。
8. 如何通过审计和批次机制控制后台操作与设备授权边界。

## 9. 当前系统边界

为了避免把系统描述得过宽，以下内容不属于当前已经完成的核心功能：

1. 面向终端用户的聊天助手功能。

3. 完整的 OTA 升级平台。
4. 默认启用的长期健康时序分析平台。
5. 超出 Beacon 消息调度范围的大型综合业务系统。

当前系统最准确的定位，仍然是：

一套面向 Beacon 终端的消息发布、设备管理、状态观测和行为回传系统。

BeaconOps 当前已经具备一套完整、可运行、跨设备端与服务器端闭环的系统能力。

它不是只有前端页面，也不是只有设备样机，而是已经把以下关键链条串起来了：

1. 设备自动接入。
2. 设备授权控制。
3. 消息创建与多目标下发。
4. recipient 级状态追踪。
5. ACK 可靠回传。
6. 健康状态回传。
7. 行为数据回传与图表展示。
8. 批次、设备、管理员、审计的后台管理。
9. 基于 SSE 的前端实时刷新。

所以从系统功能层面看，BeaconOps 已经是一套完成度较高的 Beacon 运维与消息控制系统，而不是概念演示或单点原型。
