# Firmware

The BeaconOps firmware runs on **ESP32-C3** with **ESP-IDF v5.x** and only **4 MB of flash**, so the top-level app has to fit under 2.87 MB — things are a bit tight. The UI side uses **LVGL 9.3** on top of FreeRTOS.

> Do not casually swap in a larger flash variant. The partition table, fonts, and SPIFFS layout are all tuned for 4 MB; changing the size means redoing the math.

## Partition Table

```
Name        Type    SubType   Offset      Size
nvs         data    nvs       0x9000      24 KB    NVS key-value storage
phy_init    data    phy       0xF000       4 KB    RF calibration data
factory     app     factory   0x10000   2.87 MB    Application firmware
storage     data    spiffs    (auto)       1 MB    SPIFFS filesystem
```

The current application binary is roughly **2.4 MB**. The `factory` partition provides 2.87 MB, leaving about 470 KB of headroom.  
If you add features or new assets, check flash usage and adjust `partitions.csv` when necessary.

```
│   └── settings/   Persistent application settings
│
├── Protocol stack

│   ├── msg/        Downlink parsing and LRU deduplication; warn/emergency are force-set to "requires acknowledgement" at parse time

│   ├── event/      Event/anomaly reporting (parse_reject, store_full, ack_give_up, etc.)
│   └── health/     Periodic health reporting (default 30 s; extra packet on battery level or charging state change)
│
├── Application layer
│   ├── profile/    Rolling 60-second behavior aggregation; windows accumulated offline are written to SPIFFS and uploaded after reconnection
│   ├── notify/     Prompt tones for all four message levels (info / notice / warn / emergency) plus confirmation tone
│   └── ui/         Screen UI and message card stack (LVGL); default font is a custom MiSans Bold GB2312 bitmap font generated by generate_misans_font.py and compiled directly into the firmware
│
└── README.md
```

## Startup Flow and Task Model

`main/main.cpp` is the assembly entry point. It wires components together, but most business logic lives in `components/`.

Startup is split into four stages:

1. **Synchronous hardware and local-service initialization**  
   GPIO, I2C, IMU, LVGL, backlight, NVS, settings, SPIFFS, audio, CW2017, and power-management setup are initialized in order.

2. **Synchronous creation of local runtime handles**  
   `error`, `notify`, `ui`, and `wifi` handles are created. Wi-Fi is only prepared here; connection is not performed on the main thread.

3. **Background `net_bringup_task` brings up networking**  
   The task runs `wifi -> sntp -> identity/hmac -> mqtt -> tx/msg/profile` in sequence. Local UI continues to work even if networking fails.

4. **Independent `sensor_task` handles behavior input**  
   IMU samples are converted into motion, shake, and behavior callbacks, then forwarded to `msg` and `profile`.

In short:

- `main/main.cpp` handles initialization and wiring
- `components/` implements functionality
- FreeRTOS tasks handle runtime scheduling

## Message Handling and Confirmation Flow

Incoming downlink messages are parsed by `msg/`:

- `info` / `notice`: marked as received immediately after parsing; no user action needed
- `warn` / `emergency`: force-set to "requires acknowledgement" at parse time; not considered confirmed until the user shakes the device

Shake callback chain: `sensor_task` detects the gesture → `msg_on_shake` → `tx/` publishes the ACK

The ACK is written to an NVS ring buffer and retried with exponential backoff. When retries are exhausted, `tx/` does not silently discard — it reports an `ack_give_up` event via `event/`, which the server can observe as a failed confirmation record for that device.

## Device Identity and Authentication

Each device derives its `device_id` from the eFuse MAC address at runtime (12 hex characters). No per-device ID needs to be flashed manually.

MQTT authentication uses shared batch credentials (`BATCH_UUID` / `BATCH_SECRET`) embedded at build time. The derived password format is:

```
<timestamp>:<nonce>:<HMAC-SHA256(batch_secret, device_id|ts|nonce)>
```

See `components/identity/identity.h` for details.

## TLS Certificate (Required)

Place the broker CA certificate in `components/certs/` as `isrg_root_x1.pem`. The build system embeds it through `EMBED_TXTFILES`.

If your broker uses **Let's Encrypt**, download ISRG Root X1 from:

```
https://letsencrypt.org/certificates/
```

Save the PEM file as `components/certs/isrg_root_x1.pem`.

If your broker uses a different CA, replace the file with the appropriate root certificate. No code change is required.

## Credential Configuration

Before building, update `components/config/config.h`:

```c
#define BATCH_UUID    "YOUR_BATCH_UUID"
#define BATCH_SECRET  "YOUR_BATCH_SECRET"

#define WIFI_CRED_LIST { \
    {"YOUR_SSID", "YOUR_PASSWORD"}, \
    {NULL, NULL} \
}
```

Also confirm the MQTT broker address:

```c
#define BROKER_URI   "mqtts://YOUR_BROKER_HOST:8883"
```

**Do not commit real credentials to version control.**
