# BeaconOps 前端规格 v1

> 状态:**权威**
> 配套:[protocol-v1.md](./protocol-v1.md)、[data-model-v1.md](./data-model-v1.md)
> 心智模型:**工作流(以"发消息"为主线)**,而不是运营面板。当前 cocandy-editorial-v2 / beacon-console 几乎要重做。

---

## 0. 核心信条

1. **首屏 = 发消息**。不是仪表盘、不是设备清单。打开 → 5 秒内就能下发一条消息。
2. **设备只是消息的目标**,不是首要对象。设备列表是"选目标"的辅助组件,不是顶层导航。
3. **每条消息从创建到终态可被完整观测**。前端不许折叠 9 状态,不许隐藏 attempt_count / last_error。
4. **LLM 与表单等价并存**。LLM 触发的下发,前端 UI 表现与人工下发完全一致(同一条 `messages` 行 + 同一个状态时间线)。
5. **没有"持有人"页面**。设备的 dept/roles/alias 是设备本身的属性,在设备详情里就地编辑。

## 2. 发消息(/send)— 主工作流

### 2.1 布局(单页,无路由跳转)

```
┌─────────────────────────────────────────────────────────────────┐
│  选目标                                                          │
│  ○ 单台设备   ○ 部门   ○ 角色   ○ 全员                          │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ [搜索设备 alias / device_id / dept ...]                 │  │
│  │ ┌──────────────────────────────────────────────────────┐│  │
│  │ │ ☑ a1b2c3d4e5f6 │ 张三的笔 │ 急诊  │ medic │ 🟢 78% │ ││  │
│  │ │ ☐ b2c3d4e5f6a7 │         │ 急诊  │       │ ⚫     │ ││  │
│  │ └──────────────────────────────────────────────────────┘│  │
│  │ 已选 1 台                                               │  │
│  └──────────────────────────────────────────────────────────┘  │
├─────────────────────────────────────────────────────────────────┤
│  写内容                                                          │
│  级别  [ info ▾ ]   TTL  [ 3600 s ] (emergency 自动永不过期)    │
│  标题  [____________________________]                          │
│  正文  [____________________________________________]          │
│        [____________________________________________]          │
│  确认方式  ( ) 不需要 (●) 收到回执 ( ) 已显示 ( ) 摇一摇确认     │
│            ↑ warn/emergency 强制锁定为"摇一摇确认"              │
├─────────────────────────────────────────────────────────────────┤
│  预览                                  [ 立即发送 ] [ 定时... ] │
│  ┌──────────────────────────┐                                  │
│  │  ⚠ info / 1h             │                                  │
│  │  开会通知                 │                                  │
│  │  下午3点到3楼会议室        │                                  │
│  └──────────────────────────┘                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 行为

- 选目标变更时,**实时显示"该规则会命中 N 台在线设备"**(`GET /devices?...&online=1` 计数)。
- 点 **立即发送** → `POST /messages` → 跳到 §3 该消息详情页,顶部高亮"已下发,跟踪中"。
- 点 **定时** → 弹出 fire_at 选择 → `POST /scheduled`。
- 选 emergency 时:TTL 输入框灰显并显示"永不过期",ack 锁死为 acknowledged。

### 2.3 字段校验(前端先校,服务器再校)

- title 与 body 至少一个非空
- level ∈ {info, notice, warn, emergency}
- 单播时 device_id 必须存在 + enabled=1
- 广播时命中数为 0 → 提示但允许提交(可能是离线场景)

## 3. 消息(/messages)

### 3.1 列表

```
┌──────────────────────────────────────────────────────────────────┐
│  [状态: 全部▾] [级别▾] [时间: 24h▾] [目标: 全部▾]      [清空]   │
├──┬──────────┬──────┬──────────┬─────┬──────┬──────────────────┤
│  │ 时间      │ 级别 │ 目标      │ 状态 │ 进度  │ 标题           │
├──┼──────────┼──────┼──────────┼─────┼──────┼──────────────────┤
│  │ 12:03:21 │ warn │ 部门:急诊 │ 派发中│ 3/5  │ 抢救准备       │
│  │ 12:00:45 │ info │ 全员      │ 已送达│ 12/12│ 开会通知       │
│  │ 11:58:09 │ emer │ 张三的笔  │ 失败  │ 0/1  │ ...            │
└──┴──────────┴──────┴──────────┴─────┴──────┴──────────────────┘
```

- "进度"列 = `acknowledged or displayed or delivered / total recipients`
- 行实时刷新(SSE `message.status_change`)
- 失败行红色,鼠标悬停显示 `last_error`

```
┌──────────────────────────────────────────────────────────────────┐
│  会话: 默认 ▾                                       [新建会话]    │
├──────────────────────────────────────────────────────────────────┤
│  你: 给急诊所有人发"3点抢救演练"                                 │
│  助手: 我将向部门=急诊 的 5 台设备发送 warn 级消息。              │
│        > tool_call: send_message(target=dept:急诊, level=warn,   │
│          title=演练通知, body=3点抢救演练)                       │
│        ⚠ 需要确认 (5 分钟内有效)  [确认] [取消]                  │
│  你: [确认]                                                       │
│  助手: 已下发,跟踪: /messages/5af9...                             │
└──────────────────────────────────────────────────────────────────┘
```

- 工具调用产生的消息 → 真实写入 `messages` 表 → 前端 SSE 收到 `message.status_change` → 在"消息"页正常出现
- 高危(emergency / 撤销批次 / 删操作员)走 `confirm_tokens` 二次确认按钮

## 6. 设置(/settings)

子标签:

| 子页 | 路由 | 内容 |
|---|---|---|
| 批次 | `/settings/batches` | 批次列表 + 创建 + 撤销 + 轮换 secret + 下载工厂烧录 CSV(`batch_uuid,batch_secret`) |
| 操作员 | `/settings/admins` | admins 增删改(仅 role=admin 可见) |
| 审计 | `/settings/audit` | audit_log 筛选与导出 |
| 关于 | `/settings/about` | 服务器版本 / 协议版本 |

## 7. 实时连接

- 入口:登录后建立单条 `EventSource('/api/v1/stream')`
- 全局 Pinia store 接收事件,广播给关心的页面(消息列表/详情、设备列表/详情)
- 断线自动重连(exp backoff 1→30 s)

---

## 8. 技术栈(建议)

- Vue 3 + `<script setup>` + TypeScript
- Vite + Element Plus(已在用)
- Pinia(实时状态)
- 路由:Vue Router,5 个顶层路由如 §1
- 一份 `api.ts` 封装所有 REST + 一份 `stream.ts` 管 SSE

> 现有 `tests/beacon-console/` 与 `tests/cocandy-editorial-v2/` 中可复用的:axios 封装、Element 主题、登录页骨架。其它(Dashboard、Holders 页、status 折叠组件)直接删除。

---

## 9. 删除清单(以前有,新版没有)

- "持有人" / "Holders" / "用户管理" 任意页面
- Dashboard / 概览 / 大屏 任意页面
- "TTS 凭据" / "音频测试" 任意入口
- 把消息状态折叠成 3 类的组件(`groupStatus` / `confirmed/unconfirmed/failed`)
- 任何手工录入 MAC 之外身份(UUID 烧录工具页)
