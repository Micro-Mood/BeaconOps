# 实际消息发送与 ACK 确认流程

> 本文按当前代码行为整理，专门描述系统现在实际上怎么跑。
> 它参考 [流程.md](流程.md) 的写法，但内容以当前实现为准，不代表理想方案。

## 一、服务端消息发送

发送一条消息之后，服务端当前实际流程如下：

```
┌─ 创建消息
│   ├─ 校验 target_kind / level / ack_mode / title / body
│   ├─ warn / emergency 强制 ack = acknowledged
│   ├─ emergency 强制 ttl = 0
│   ├─ 展开收件人（只取 enabled=1 的设备）
│   ├─ 写 messages.status = queued
│   ├─ 写 message_recipients.status = queued
│   └─ 启动异步 publish 任务
│
└─ publish 任务
    ├─ MQTT publish 成功
    │   ├─ ack = none
    │   │   ├─ messages.status = delivered
    │   │   ├─ recipients.status = delivered
    │   │   └─ 不再等待设备 ACK
    │   │
    │   └─ ack != none
    │       ├─ messages.status = sent
    │       ├─ recipients.status = sent
    │       ├─ 写 next_retry_at
    │       └─ 等待设备上报 ACK 推进后续状态
    │
    └─ MQTT publish 失败
        ├─ attempt_count + 1
        ├─ last_error = mqtt_publish_failed
        └─ next_retry_at = backoff 后的下次重试时间
```

### 备注

- 当前实现不会先判断“在线/不在线”再走两套路径。
- 真实逻辑是“先入库，再尝试 publish，再靠重试和 ACK 收敛状态”。
- 服务端有 retry loop：
  - TTL 到期但未终态的消息会被标记为 `expired`
  - `queued/sent` 且到重试时间的消息会继续重发
- 达到最大重试次数后，消息会被标记为 `failed`，原因是 `received_ack_timeout`。

## 二、设备端消息接收与本地实例

设备端收到 `device/{id}/cmd` 或广播消息之后，当前实际流程如下：

```
┌─ parser 解析 payload
│   ├─ 非法 JSON / 缺字段 / 非法 level
│   │   └─ 上报 event(parse_reject) 并丢弃
│   │
│   └─ 合法 payload
│       └─ 进入 msg_ingest
│
└─ msg_ingest
    ├─ LRU 去重命中
    │   └─ 当前实现仍会上报一次 received
    │
    └─ 新消息
        ├─ 先写 SPIFFS 消息文件
        ├─ 再尝试进入内存消息队列
        ├─ 成功接纳
        │   └─ 上报 received
        │
        └─ admit 失败但已落盘
            ├─ 仍保留本地文件
            └─ 超时场景下可转为 expired
```

### 实际存的不是 ACK 队列，而是消息实例文件

SPIFFS 当前保存的是消息实例，不是旧草稿里的“SPIFFS ACK 队列”。

| 后缀 | 含义 |
|------|------|
| `.j` | 已接收，仍处于待显示 / 待确认状态 |
| `.a` | 本地已判定为 acknowledged，等待 ACK 真正送达服务端 |
| `.e` | 本地已判定为 expired，等待 ACK 真正送达服务端 |

### 备注

- 当前只接受落地协议里的消息格式：`id / ts / ttl / level / title / body / ack`。
- 不再兼容旧的嵌套 display、整数 level、音频字段等旧格式。
- 当前代码里 `received` 的发送比协议语义更宽：
  - 去重命中会发一次 `received`
  - 新消息成功落盘并接纳后也会发一次 `received`
  - 这个动作当前没有在 `msg` 层按 `ack_mode` 严格过滤

## 三、设备端显示、摇一摇与过期

设备端消息进入 `msg` 调度器之后，当前实际流程如下：

```
轮询调度器（默认 100 ms tick）
  ├─ step_sweep
  │   └─ 清掉已过期的非当前消息，必要时产出 expired
  │
  ├─ step_current
  │   ├─ 当前卡片首次上屏
  │   │   ├─ ui_push_card
  │   │   ├─ notify_play
  │   │   ├─ 紧急级别强制亮屏
  │   │   └─ 若 ack_mode >= displayed，则上报 displayed
  │   │
  │   └─ 当前卡片 TTL 到期
  │       ├─ 本地文件改为 .e
  │       ├─ 产出 expired
  │       └─ dismiss 当前卡片
  │
  ├─ step_shake
  │   ├─ 满足 min_display_ms 后才允许确认
  │   ├─ 若 ack_mode >= acknowledged，则上报 acknowledged
  │   ├─ 本地文件改为 .a
  │   └─ dismiss 当前卡片
  │
  └─ step_pick
      ├─ 当前无卡片时，挑最高优先级消息显示
      └─ 若出现更高优先级消息，可在最小显示时长后抢占
```

### 备注

- 当前默认 `min_display_ms = 2000`。
- `displayed` 只会在首次上屏时发一次。
- 紧急消息不会因为 TTL 自动过期。
- “用户已经摇一摇确认”不等于“本地文件已经彻底删除”。

## 四、设备端 ACK 上报流程

当前实现不是在 `msg` 组件里直接发 MQTT ACK，而是通过 `tx` 组件做持久化重发：

```
msg 组件产出 ACK
  └─ cb_msg_ack
      └─ tx_emit_ack
          ├─ 先写 NVS pending ring
          ├─ MQTT 在线时尽力立即 flush
          └─ MQTT 不在线时等待后台 drain task
```

### NVS pending ring

ACK 待发送队列保存在 NVS，而不是 SPIFFS。每条记录的核心字段如下：

| 字段 | 说明 |
|------|------|
| `id` | 消息 ID |
| `kind` | ACK 类型 |
| `attempts` | 已重试次数 |
| `failed` | 是否达到上限后停发 |
| `ts` | ACK 产生时间 |
| `next_due_ms` | 下次可重试时间 |

ACK 类型如下：

| ACK类型 | 含义 |
|---------|------|
| `received` | 设备已接触到消息 |
| `displayed` | 消息首次上屏 |
| `acknowledged` | 用户已摇一摇确认 |
| `expired` | 消息在设备端超时过期 |

### 响应机（ACK 上报）

设备端当前会按 NVS pending ring 的顺序尝试上报 ACK：

```
轮询 NVS pending ring
  └─ 存在未完成的 ACK
      ├─ 到期可发送
      │   ├─ publish 成功
      │   │   ├─ 删除本条 ACK
      │   │   └─ 若 kind 为 acknowledged / expired
      │   │       └─ 回调删除对应 SPIFFS 消息文件
      │   │
      │   └─ publish 失败
      │       ├─ attempts + 1
      │       ├─ 指数退避后重试
      │       └─ 达到上限后标记 failed
      │
      └─ 尚未到 next_due_ms
          └─ 本轮跳过
```

### 备注

- 当前默认 `max_attempts = 10`。
- 当前默认 `drain_period_ms = 5000`。
- 当前默认 `max_pending = 32`。
- QoS 映射：
  - `acknowledged` 使用 QoS 2
  - `received / displayed / expired` 使用 QoS 1
- ACK 达上限后的真实行为不是“弹窗后删除本条 ACK”，而是：
  - 标记 `failed=1`
  - 上报一次 `ack_give_up`
  - 停止继续发送

## 五、服务端接收 ACK 后的状态推进

服务端收到 `device/{id}/uplink/ack` 后，当前实际流程如下：

```
┌─ 收到 uplink ack
│   ├─ 写 message_events
│   ├─ 更新对应 recipient 状态
│   │   ├─ received     -> delivered
│   │   ├─ displayed    -> displayed
│   │   ├─ acknowledged -> acknowledged
│   │   └─ expired      -> expired
│   │
│   └─ 刷新整条消息的聚合状态
│       ├─ 所有活动 recipient >= delivered
│       │   └─ messages.status = delivered
│       ├─ 所有活动 recipient >= displayed
│       │   └─ messages.status = displayed
│       └─ 所有活动 recipient >= acknowledged
│           └─ messages.status = acknowledged
```

### 备注

- 这里的“活动 recipient”会排除已经进入终态的 `expired / failed` 收件人。
- 多收件人消息不是任一设备确认就全局确认，而是按所有活动收件人聚合推进。
- 服务端整体状态机实际是：

```text
queued -> sent -> delivered -> displayed -> acknowledged
   |        |         |            |
   |        |         |            +-> expired
   |        |         +-> expired
   |        +-> failed
   +-> failed
```

## 六、可配置常量

| 参数 | 说明 | 默认值 |
|------|------|--------|
| 服务端消息重试次数 | 单条消息下发重试上限 | 10 |
| 设备端 ACK 重试次数 | 单条 ACK 上报重试上限 | 10 |
| 设备端 ACK 重试周期 | drain task 周期 | 5000 ms |
| ACK 队列容量 | NVS pending ring 上限 | 32 |
| 消息本地槽位上限 | SPIFFS 消息实例上限 | 64 |
| 最小显示时长 | 摇一摇确认前最短展示时间 | 2000 ms |
| 消息过期时间 | 由服务端消息 `ttl` 决定 | configurable |
