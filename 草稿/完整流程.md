# 消息发送与 ACK 确认完整流程

> 这是推荐的目标流程草稿。
> 原则：按选中设备逐台追踪；在线立即发，离线等待上线；设备端先落盘再显示，ACK 独立可靠上报。

## 一、服务端消息发送

发送一条消息之后，服务端先展开所有选中的设备，再按设备逐台处理：

```
┌─ 创建消息
│   ├─ 校验消息内容
│   ├─ 生成消息ID
│   ├─ warn / emergency 强制要求 acknowledged
│   ├─ emergency 强制不过期
│   ├─ 展开选中的设备
│   │   ├─ 单设备：1 台
│   │   ├─ 部门：该部门所有启用设备
│   │   └─ 全员：所有启用设备
│   ├─ 写入 messages
│   └─ 为每台设备写入 message_recipients
│       └─ 初始状态 queued
│
└─ 按 recipient 投递
    ├─ 设备在线
    │   ├─ 尝试发送到 device/{id}/cmd
    │   ├─ publish 成功
    │   │   ├─ recipient.status = sent
    │   │   ├─ 记录 sent_at / attempt_count
    │   │   └─ 等待设备 ACK 推进状态
    │   │
    │   └─ publish 失败
    │       ├─ 记录失败原因
    │       ├─ attempt_count + 1
    │       ├─ 未达上限：进入下次重试
    │       └─ 达到上限：recipient.status = failed
    │
    └─ 设备不在线
        ├─ 不消耗发送次数
        ├─ recipient.status 保持 queued
        └─ 等待设备上线后继续投递
```

### 备注

- 服务端不应该只按整条 message 计数，而应该按每台 recipient 计数。
- 离线不是失败；离线只是暂时不能发送。
- 广播和部门消息只是“选设备”的方式，最终仍要展开到每台设备追踪。
- `ack=none` 的消息，在线 publish 成功后可结束等待；离线设备仍要 queued 到上线后再发。

## 二、设备上线补发

设备上线后，服务端立即检查这台设备是否有待发送消息：

```
设备上线
  ├─ 更新 devices.online = 1
  ├─ 查找该 device_id 的 pending recipients
  │   ├─ status = queued
  │   └─ status = sent 但仍未收到 received
  │
  └─ 逐条处理
      ├─ 消息已过期
      │   └─ recipient.status = expired
      │
      └─ 消息未过期
          └─ 进入在线投递流程
```

### 备注

- 设备上线是补发触发点。
- 后台定时重试是兜底，不应该替代上线补发。
- 如果设备长期不上线，消息到期后进入 expired，而不是 failed。

## 三、设备端接收消息

设备收到下行消息后，先保存，再显示：

```
收到 MQTT cmd
  ├─ 解析 payload
  │   ├─ 解析失败
  │   │   ├─ 丢弃消息
  │   │   └─ 上报 event(parse_reject)
  │   │
  │   └─ 解析成功
  │       └─ 进入消息实例流程
  │
  └─ 消息实例流程
      ├─ LRU 去重
      │   ├─ 重复消息
      │   │   └─ 可补发 received，帮助服务端停止重试
      │   │
      │   └─ 新消息
      │       ├─ 先写 SPIFFS 消息实例
      │       ├─ 写入成功
      │       │   ├─ 放入内存消息队列
      │       │   └─ 写 ACK 队列：received
      │       │
      │       └─ 写入失败
      │           ├─ 重试 N 次
      │           ├─ 仍失败：上报 event(store_full / store_failed)
      │           └─ 不进入显示队列
```

### 消息实例状态

| 状态 | 含义 |
|------|------|
| live | 已收到，等待显示或等待确认 |
| acknowledged_pending | 用户已确认，等待 ACK 成功上报 |
| expired_pending | 消息已过期，等待 expired ACK 成功上报 |
| done | 最终 ACK 已送达服务端，本地可删除 |

## 四、设备端显示与确认

设备端从本地消息队列里选择当前要显示的消息：

```
消息调度器
  ├─ 当前无消息
  │   └─ 挑选最高优先级消息显示
  │
  ├─ 当前有消息
  │   ├─ 新消息优先级更高
  │   │   ├─ 当前消息已满足最小显示时长
  │   │   │   └─ 高优先级消息抢占显示
  │   │   └─ 未满足最小显示时长
  │   │       └─ 暂不抢占
  │   │
  │   └─ 当前消息继续显示
  │
  ├─ 首次上屏
  │   ├─ 点亮屏幕
  │   ├─ LVGL 渲染卡片
  │   ├─ 播放本地提示音
  │   └─ 写 ACK 队列：displayed
  │
  ├─ 用户摇一摇
  │   ├─ 已满足最小显示时长
  │   │   ├─ 状态改为 acknowledged_pending
  │   │   ├─ 写 ACK 队列：acknowledged
  │   │   └─ 从当前显示位移除，继续显示下一条
  │   │
  │   └─ 未满足最小显示时长
  │       └─ 等待满足后再确认
  │
  └─ TTL 到期
      ├─ emergency 不自动过期
      └─ 普通消息
          ├─ 状态改为 expired_pending
          ├─ 写 ACK 队列：expired
          └─ 从显示队列移除
```

### 备注

- 状态不能跳：received → displayed → acknowledged。
- 未显示的消息不能直接 acknowledged。
- 用户摇一摇只确认当前正在显示的消息。
- 紧急消息优先级最高，并且不会自动 TTL 过期。

## 五、设备端 ACK 队列与上报

设备端所有 ACK 都先进入本地 ACK 队列，再由响应机统一上报：

```
产生 ACK
  ├─ received
  ├─ displayed
  ├─ acknowledged
  └─ expired
      ↓
写入本地 ACK 队列
      ↓
响应机按时间顺序上报
```

### 响应机流程

```
轮询 ACK 队列
  └─ 存在未上报 ACK
      ├─ MQTT 在线
      │   ├─ publish 成功
      │   │   ├─ 删除本条 ACK
      │   │   └─ 若是 acknowledged / expired
      │   │       └─ 删除对应消息实例
      │   │
      │   └─ publish 失败
      │       ├─ 尝试次数 + 1
      │       ├─ 未达上限：等待下次重试
      │       └─ 达到上限
      │           ├─ 上报 event(ack_give_up)
      │           ├─ 本地提示用户或记录错误
      │           └─ 进入失败处理
      │
      └─ MQTT 不在线
          └─ 保留 ACK，等待网络恢复
```

### 备注

- ACK 队列必须持久化，设备重启后仍能继续上报。
- ACK 成功送达服务端后，才允许删除对应的最终消息实例。
- ACK 上报失败不应该影响消息状态跳跃。

## 六、服务端接收 ACK

服务端收到设备 ACK 后，只更新对应设备的 recipient 状态：

```
收到 device/{id}/uplink/ack
  ├─ 校验 msg_id / kind
  ├─ 写 message_events
  ├─ 更新该 device_id 对应 recipient
  │   ├─ received     -> delivered
  │   ├─ displayed    -> displayed
  │   ├─ acknowledged -> acknowledged
  │   └─ expired      -> expired
  │
  └─ 重新聚合整条 message 状态
      ├─ 所有未终态 recipient >= delivered
      │   └─ message.status = delivered
      ├─ 所有未终态 recipient >= displayed
      │   └─ message.status = displayed
      └─ 所有未终态 recipient >= acknowledged
          └─ message.status = acknowledged
```

### 备注

- 一个设备 ACK 只推进自己的 recipient。
- 整条 message 的状态由所有 recipient 聚合得到。
- failed / expired 的 recipient 不阻塞其他设备继续推进。

## 七、失败与过期

失败和过期分开处理：

```
设备离线
  └─ queued 等待上线，不算失败

消息 TTL 到期
  ├─ 未发送设备：recipient = expired
  ├─ 已发送但未确认设备：recipient = expired
  └─ 已确认设备：保持 acknowledged

在线发送连续失败达到上限
  └─ recipient = failed

设备端 ACK 连续失败达到上限
  ├─ event = ack_give_up
  └─ 服务端标记对应消息 / recipient 异常
```

### 备注

- 离线等待不是 failed。
- TTL 到期是 expired。
- 网络在线但连续发送失败才是 failed。

## 八、状态总览

### recipient 状态

| 状态 | 含义 |
|------|------|
| queued | 已选中，等待发送 |
| sent | 已向该设备下发，等待 received |
| delivered | 设备已收到 |
| displayed | 设备已显示 |
| acknowledged | 用户已确认 |
| expired | 已过期 |
| failed | 明确失败 |

### message 聚合状态

```
queued
  └─ sent
      └─ delivered
          └─ displayed
              └─ acknowledged

任意未完成分支可进入：
  ├─ expired
  └─ failed
```

## 九、可配置常量

| 参数 | 说明 | 默认值 |
|------|------|--------|
| 服务端在线发送重试次数 | 单台设备在线 publish 重试上限 | 10 |
| 设备端 ACK 上报重试次数 | 单条 ACK 上报失败重试上限 | 10 |
| ACK 队列容量 | 设备端本地 ACK 队列容量 | configurable |
| 消息本地容量 | 设备端本地消息实例容量 | configurable |
| 最小显示时长 | 摇一摇确认前必须显示的时间 | 2000 ms |
| 消息 TTL | 超时后进入 expired | configurable |
