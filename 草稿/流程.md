# 消息发送与 ACK 确认流程

## 一、服务端消息发送

发送一条消息之后，服务器首先判断选中的设备：

```
┌─ 选中
│   ├─ 在线
│   │   ├─ 尝试 N 次发送消息
│   │   │   ├─ 成功
│   │   │   │   └─ 记录响应（服务端侧）
│   │   │   │       ├─ 实例化 message class
│   │   │   │       │   ├─ 消息ID
│   │   │   │       │   ├─ 消息收到时间
│   │   │   │       │   └─ 消息状态: 收到 → 显示 → 确认
│   │   │   │       │
│   │   │   │       ├─ 状态为 收到 → 写设备端 ACK 队列 (ackrecv)
│   │   │   │       │
│   │   │   │       ├─ 屏幕亮，LVGL 渲染卡片
│   │   │   │       │   ├─ 主页改为插槽模式
│   │   │   │       │   ├─ 一个实例一个卡片，自动排列
│   │   │   │       │   ├─ 紧急消息置顶
│   │   │   │       │   └─ 同类型按时间最早靠前
│   │   │   │       │   └─ 状态改为 显示 → 写 ACK 队列 (ackshow)
│   │   │   │       │
│   │   │   │       └─ 用户摇一摇
│   │   │   │           └─ 状态改为 确认 → 写 ACK 队列 (ackshake)
│   │   │   │
│   │   │   └─ 失败
│   │   │       ├─ 记录失败日志
│   │   │       └─ 标记该设备发送失败
│   │   │
│   │   └─ 备注
│   │       ├─ 所有实例不直接响应，先写 SPIFFS
│   │       ├─ 写入失败重试 N 次 → 仍失败弹窗提醒
│   │       ├─ 状态不能跳：收到 → 显示 → 确认（不能直接跳到确认）
│   │       └─ 未确认消息持久化 NVS，ackshake 成功后彻底删除
│   │
│   └─ 不在线
│       └─ 存入缓冲区 → 等待上线后进入在线流程
│
└─ 未选中
    └─ 不处理
```

## 二、SPIFFS ACK 队列

采用 JSON 格式存储：

| 字段 | 说明 |
|------|------|
| 消息ID | 消息唯一标识 |
| 时间 | 记录时间戳 |
| ACK类型 | ackrecv / ackshow / ackshake |
| 尝试次数 | 当前重试计数 |

ACK 类型定义：

| ACK类型 | 含义 |
|---------|------|
| ackrecv | 设备已收到消息 |
| ackshow | 设备已显示消息 |
| ackshake | 用户已摇一摇确认 |

## 三、响应机（ACK 上报）

设备端将 SPIFFS 中的 ACK 记录按时间顺序上报（旧优先）：

```
轮询 SPIFFS 队列
  └─ 存在未上报的 ACK
      ├─ 尝试 N 次上报
      │   ├─ 成功 → 删除本条 ACK
      │   └─ 全部失败
      │       ├─ 弹窗警告
      │       └─ 删除本条 ACK（放弃上报）
      └─ 无未上报 → 继续等待
```

## 四、可配置常量

| 参数 | 说明 | 默认值 |
|------|------|--------|
| 消息推送重试次数 | 服务端单条消息推送重试上限 | 10 |
| ACK上报重试次数 | 设备端上报失败重试上限 | 10 |
| SPIFFS写入重试次数 | 各阶段写入重试上限 | 3 |
| 消息过期时间 | 超时未确认则自动丢弃 | configurable |
