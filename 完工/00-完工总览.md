## 2. 当前系统的完工结论

### 2.1 总体结论

BeaconOps 当前已经形成一条可闭环运行的消息调度链路：

1. 固件端以设备 MAC 作为 device_id，使用批次凭证完成 MQTT HMAC 接入鉴权。
2. 服务端以 FastAPI + SQLite + gmqtt 组成业务核心，完成批次、设备、消息、管理员、审计、SSE 实时推送与行为数据聚合。
3. 前端控制台已经具备登录、消息发送、消息历史、设备、批次、设置等完整页面，并已经接入设备行为图。
4. 协议层已稳定为 cmd / ack / event / health / profile 五类消息与 topic 体系。

### 2.2 当前真正落地的核心能力

| 方向 | 当前结论 |
|---|---|
| 硬件与固件 | 已实现显示、提示音、IMU 行为采样、健康上报、消息接收、摇一摇确认、ACK 持久重发、行为窗口上报 |
| 服务器 | 已实现 JWT 管理员认证、设备接入鉴权、批次白名单、消息投递与重试、消息状态推进、审计、SSE |
| 前端 | 已实现控制台登录、发送、历史、设备详情、批次详情、管理员管理、审计查询、设备行为图 |
| 协议 | 已实现基于 MQTT 的 cmd / ack / event / health / profile 协议闭环 |

### 2.3 当前已确认的系统边界

以下边界已经由现有实现坐实：

| 项目 | 当前状态 |
|---|---|
| TTS / 音频流下发 | 未进入当前业务链路；协议与服务端均未保留音频 topic |
| Holder / User 业务对象 | 已删除；当前业务对象只有 batch、device、message、admin |
| 设备 OTA | 当前文档与实现均未形成上线能力 |
| 前端 LLM 助手页 | 当前控制台未实现，现有路由中不存在 chat 页面 |
| WebSocket | 当前控制台实时更新采用 SSE，而不是 WebSocket |
| 设备部门广播订阅 | 协议和服务端支持，但固件当前装配中尚未把 dept 注入 MQTT 客户端 |

## 4. 当前实现与旧文档的差异结论

本批完工文档不再把旧论文稿当作“当前系统说明书”，原因如下：

1. 论文稿中包含 chat / MCP / TTS / holders / telemetry 等更早期或更宏观的设计。
2. 落地稿已经明确删除了 holder、TTS、旧 telemetry 与部分旧 topic。
3. 当前真实实现进一步收敛成“设备消息调度系统”，并已在行为图、SSE、批次管理、设备行为聚合上继续演进。

因此，本文件夹采用的判断优先级为：

1. 当前真实实现
2. 落地文档
3. 论文与草稿文档
