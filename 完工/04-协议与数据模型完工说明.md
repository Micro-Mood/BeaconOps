## 2. 当前身份模型

### 2.1 当前三层身份结构

| 层级 | 当前实体 | 说明 |
|---|---|---|
| 管理面 | admin | 只存在于控制台与服务端 |
| 批次面 | batch_uuid | 代表一组设备的签发域与容量边界 |
| 设备面 | device_id | 由芯片 eFuse MAC 派生的 12 位十六进制标识 |

### 2.2 当前设备 MQTT 登录模型

| MQTT 字段 | 当前值 |
|---|---|
| client_id | device_id |
| username | batch_uuid |
| password | ts:nonce:hmac_hex |

### 2.3 当前 HMAC 摘要输入

服务端和固件当前都围绕同一条规则工作：

1. 输入 device_id。
2. 输入当前 unix 时间 ts。
3. 输入随机 nonce。
4. 使用 batch_secret 做 HMAC-SHA256。

### 2.4 当前身份模型带来的边界

1. 设备本身不保存服务器下发的长期 token。
2. 批次撤销就意味着该批次下所有设备在协议层失效。
3. device_id 的来源是硬件 MAC，因此设备号天然和芯片绑定。
4. 批次容量是服务端鉴权过程中的强约束，而不是事后统计。

## 3. 当前 MQTT topic 体系

### 3.1 服务端下行 topic

| topic | 方向 | 当前用途 |
|---|---|---|
| device/{device_id}/cmd | 服务端 -> 单设备 | 单播下发消息 |
| broadcast/all/cmd | 服务端 -> 所有设备 | 全员广播 |
| broadcast/dept/{dept}/cmd | 服务端 -> 指定部门设备 | 部门广播 |

### 3.2 设备上行 topic

| topic | 方向 | 当前用途 |
|---|---|---|
| device/{device_id}/status | 设备 -> 服务端 | 在线状态与版本信息 |
| device/{device_id}/uplink/ack | 设备 -> 服务端 | received / displayed / acknowledged / expired |
| device/{device_id}/uplink/event | 设备 -> 服务端 | 端侧异常与丢弃事件 |
| device/{device_id}/uplink/health | 设备 -> 服务端 | 设备健康快照 |
| device/{device_id}/uplink/profile | 设备 -> 服务端 | 60 秒行为窗口 |

### 3.3 当前 retain 与 QoS 结论

| 消息类型 | QoS / retain |
|---|---|
| status online | QoS 1，retain=true |
| status offline LWT | QoS 1，retain=true |
| cmd | QoS 1，retain=false |
| ack acknowledged | QoS 2 |
| ack 其他种类 | QoS 1 |
| event | QoS 1 |
| health | QoS 0 |
| profile | QoS 1 |

### 3.4 当前实现边界

虽然协议支持 broadcast/dept/{dept}/cmd，但固件当前装配并未把 dept 注入 MQTT 配置，因此“部门广播”在协议和服务端已存在，在当前固件运行装配上仍是未闭环状态。

## 4. 当前下行命令 payload

### 4.1 当前 cmd payload 字段

| 字段 | 含义 |
|---|---|
| id | 服务端分配的消息 ID |
| ts | 服务端发送时间 |
| ttl | 生存时长，0 表示不过期 |
| level | info、notice、warn、emergency |
| title | 标题 |
| body | 正文 |
| ack | none、received、displayed、acknowledged |

### 4.2 当前 parser 约束

固件 parser 当前只接受扁平 schema，不再接受旧版嵌套结构或音频字段链路。

### 4.3 当前强制改写规则

| 条件 | 当前行为 |
|---|---|
| level = warn | ack 强制改为 acknowledged |
| level = emergency | ack 强制改为 acknowledged，ttl 强制改为 0 |
| title 与 body 同时为空 | 拒收 |
| level 非法 | 拒收 |
| ack 非法 | 拒收 |

### 4.4 当前未进入协议主链的旧字段

以下字段在旧稿中出现过，但当前实现不应再被描述为有效业务字段：

1. audio_text 业务字段。
2. 音频 topic。
3. TTS 或 PCM 相关下行字段。

## 5. 当前 ACK 协议

### 5.1 ACK 类型

| kind | 含义 |
|---|---|
| received | 已接纳到本地队列或持久化区 |
| displayed | 首次真正上屏 |
| acknowledged | 用户摇一摇确认 |
| expired | TTL 到期未确认 |

### 5.2 ACK 上行 payload 字段

| 字段 | 含义 |
|---|---|
| msg_id | 对应消息 ID |
| kind | ACK 类型 |
| ts | 设备侧时间戳 |

### 5.3 当前 ACK 与 ack_mode 的关系

ack_mode 是服务器对设备提出的“至少汇报到哪一步”的要求；ACK kind 是设备实际发生的阶段事件。

| ack_mode | 当前要求 |
|---|---|
| none | 不要求设备上报 ACK |
| received | 至少汇报 received |
| displayed | 至少汇报 displayed，隐含前一步 |
| acknowledged | 直到 acknowledged，隐含前两步 |

### 5.4 当前 ACK 可靠性机制

| 项 | 当前实现 |
|---|---|
| 本地持久化 | NVS |
| 命名空间 | tx_pending |
| 最大待发数 | 32 |
| 单轮最大发送数 | 4 |
| 基础退避 | 2000 ms |
| 最大退避 | 300000 ms |
| 最大重试次数 | 10 |

### 5.5 ACK 失败后的协议外补偿

当 ACK 达到上限仍发送失败时，设备不会静默吞掉，而是会上报一条 kind=ack_give_up 的 event，并把失败的 ACK 类型写入 reason。这样服务端至少能知道“ACK 业务链断在了哪里”。

## 6. 当前 event 协议

### 6.1 event 类型

| kind | 当前语义 |
|---|---|
| ack_give_up | ACK 连续重试失败到上限 |
| parse_reject | 收到非法命令 payload |
| store_full | 本地存储或缓冲区满 |
| dedup_drop | 因去重命中而丢弃 |

### 6.2 event payload 字段

| 字段 | 说明 |
|---|---|
| kind | 事件类型 |
| msg_id | 可选；与具体消息相关时携带 |
| reason | 可选；用于写失败原因或细节 |
| ts | 设备侧时间戳 |

### 6.3 当前 event 的可靠性结论

当前 event 组件采用 RAM 队列，长度为 8。它会等 MQTT 可用后一段时间再尝试发送，但没有像 ACK 一样做 NVS 级长期持久化。因此 event 更适合作为告警补充，而不是唯一审计来源。

## 7. 当前 health 协议

### 7.1 health payload 字段

| 字段 | 说明 |
|---|---|
| fw | 固件版本 |
| batch | 批次标识 |
| ts | 设备侧时间戳 |
| battery | 电量百分比 |

| ack_pending | ACK 待发数量 |

### 7.2 当前 health 发送规则

| 项 | 当前值 |
|---|---|
| 默认周期 | 30 秒 |
| 电量变化阈值 | 3% |

### 7.3 当前 health 数据在服务端的去向

| device_health_history 表 | 只有 HEALTH_HISTORY_ENABLED 打开时才写入 |
| SSE | 发出 device.health 事件，供前端刷新 |

## 8. 当前 profile 协议

### 8.1 profile payload 字段

| 字段 | 说明 |
|---|---|
| ts | 窗口结束时的 unix 秒 |
| win | 窗口长度，当前为 60 |
| static | 静止秒数 |
| walk_slow | 低幅活动秒数 |
| walk_fast | 中幅活动秒数 |
| run | 高幅活动秒数 |
| shake_or_fall | 剧烈波动秒数 |
| shake_n | 摇晃次数 |
| steps | 当日步数累计，未准备好时可不带 |
| intensity_avg | 平均强度 |

### 8.2 当前窗口与离线补偿

| 项 | 当前实现 |
|---|---|
| 聚合窗口 | 60 秒 |
| 默认离线队列上限 | 200 条 |
| 默认单次补发量 | 8 条 |
| 文件命名方式 | profile_q_{ts}.json |
| 补发时机 | 新窗口发布成功后顺序补发旧文件 |

### 8.3 当前 profile 在服务端的落地方式

服务端收到 profile 后会：

1. 校验设备是否仍启用且批次未撤销。
2. 按 device_id + ts 唯一键写入 behavior 表。
3. 更新 devices.last_seen_at。
4. 触发 device.behavior SSE 事件。

这正是当前前端设备行为图的数据基础。

## 9. 当前 status 协议

### 9.1 status 在线消息

设备建立 MQTT 连接成功后，会立即发布在线状态，当前在线 payload 至少包含：

1. online=true
2. fw
3. batch
4. ts

### 9.2 status 离线消息

MQTT last will 当前会在掉线时发布一条最小离线消息，只包含 online=false。

### 9.3 status 在服务端的作用

服务端收到 status 后，会更新：

1. online
2. fw_version
3. last_seen_at
4. updated_at

并向前端发出 device.online 或 device.offline。

## 10. 当前消息状态机

### 10.1 recipient 级状态机

| 状态 | 当前含义 |
|---|---|
| queued | 已落库，尚未发送 |
| sent | 已投递到 broker，等待设备 ACK |
| delivered | 不要求 ACK 或已视为送达 |
| displayed | 设备确认已显示 |
| acknowledged | 用户已确认 |
| expired | TTL 到期 |
| failed | 超过重试上限或发布失败 |

### 10.2 message 聚合状态机

messages 表中的 status 是对所有 recipients 的聚合结果，服务端会在每次 recipient 推进后重新刷新聚合态，用于历史页与详情页展示。

### 10.3 设备在线状态机

| 状态源 | 当前影响 |
|---|---|
| status online=true | 置设备在线 |
| status online=false 或 LWT | 置设备离线 |
| 批次撤销或设备禁用后仍上行 | 服务端强制改为离线并拒收后续上行 |

## 11. 当前数据落库映射

### 11.1 协议消息与数据库表的对应关系

| 协议类型 | 主落表 | 辅助影响 |
|---|---|---|
| cmd | messages、message_recipients | 触发 message.created SSE |
| ack | message_recipients、messages、message_events | 触发 recipient_update、status_change、message.event |
| event | message_events | 触发 message.event |
| health | devices，条件成立时写 device_health_history | 触发 device.health |
| profile | behavior | 触发 device.behavior |
| status | devices | 触发 device.online 或 device.offline |

### 11.2 当前前端图表数据链

设备行为图的完整链路如下：

1. sensor_task 每秒给出行为状态。
2. profile 组件按 60 秒窗口聚合。
3. 设备通过 uplink/profile 上报。
4. 服务端写入 behavior 表。
5. 服务端通过 /devices/{id}/behavior 按窗口聚合返回。
6. 前端 DeviceBehaviorPanel 绘制图表并订阅 device.behavior 自动刷新。

这条链路已经跨固件、服务端、前端三端闭环。

## 12. 当前协议边界与非现状项

### 12.1 当前不应写成“已实现协议能力”的内容

1. 音频流或 TTS 下发协议。
2. 面向 AI 助手、知识库、聊天会话的独立 topic。
3. 设备 OTA 控制协议。
4. holder/user 维度的业务对象协议。

### 12.2 当前确实存在但要谨慎描述的内容

1. 部门广播 topic 已定义，服务端 ACL 也支持，但固件装配未闭环。
2. device_health_history 表已存在，但默认不开启写入。
3. profile steps 字段是“当日步数累计”，不是窗口内步数增量。
4. status 离线消息是 LWT 最小包，不等于完整 health 快照。

## 13. 协议与数据模型完工结论

当前 BeaconOps 协议已经稳定收敛为一套非常清晰的消息调度协议：

1. 身份层使用 device_id + batch_uuid + HMAC password。
2. 下行层只保留 cmd 一种业务载荷，字段扁平，规则明确。
3. 上行层按 status、ack、event、health、profile 五类拆分职责。
4. 数据库层以 batches、devices、messages、message_recipients、message_events、behavior 为主骨架。
5. 前端设备行为图、历史状态展示、设备在线状态都能从这套协议中直接取数。

因此，当前系统的协议面已经足够支撑实际消息下发、设备观测和行为可视化，而不是停留在实验性定义阶段。
