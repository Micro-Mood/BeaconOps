## 2. 当前前端技术形态

### 2.1 当前已确认技术栈

| 维度 | 当前实现 |
|---|---|
| 框架 | Vue 3.5 |
| 语言 | TypeScript |
| 构建工具 | Vite 6 |
| 路由 | Vue Router 4 |
| 状态管理 | Pinia 2 |
| UI 基础库 | Element Plus 2.9 |
| HTTP 客户端 | axios |
| 图表 | ECharts 5.5 + vue-echarts 7 |

### 2.2 当前 API 接入方式

| 项 | 当前实现 |
|---|---|
| API 基础前缀 | /beacon/api/v1 |
| Cookie 策略 | withCredentials = true |
| CSRF 头 | 请求拦截器从 beacon_csrf 读取后写入 X-CSRF-Token |
| 登录失效处理 | 响应拦截器遇到 401 自动跳回登录页 |

### 2.3 当前本地开发旁路能力

当前 auth store 中存在开发环境旁路登录逻辑，但满足三个条件才生效：

1. 处于 Vite 开发模式。
2. VITE_BYPASS_LOGIN=true。
3. 当前 hostname 是 localhost、127.0.0.1 或 ::1。

因此，该旁路能力不是生产运行特性，而是仅用于本地开发调试。

## 3. 当前权限模型

### 3.1 前端角色结论

| 角色 | 前端能力 |
|---|---|
| admin | 可发送消息、可修改对象、可管理设置 |
| operator | 可发送消息、可修改设备和批次、不可管理管理员设置 |
| viewer | 只读访问，不可发送、不可修改 |

### 3.2 当前前端权限判断

前端权限不是写死在每个按钮里，而是由 auth store 统一给出三个布尔能力：

1. canSendMessages
2. canManageSettings
3. canMutate

这使页面侧判断比较统一，例如：

1. Send 页会在 viewer 角色下阻止真正发送。
2. DeviceDetail 页会在 viewer 角色下禁止修改 alias 与 dept。
3. Admins 设置页会在非 admin 角色下直接显示“没有权限”。

### 4.2 已确认不存在的页面

以下页面在旧前端规格里出现过，但在当前真实路由中不存在：

1. Chat 助手页。
2. LLM 对话页。
3. 单独的系统监控大盘页。
4. 面向终端用户的独立门户页。

因此，当前前端必须被描述为“消息发布与设备管理控制台”，而不是“带 AI 助手的综合操作台”。

## 5. 当前页面功能清单

### 5.1 首页 Home

当前首页是一个轻量入口页，已确认具备：

1. 标题 BeaconOps。
2. 一句产品定位文案。
3. 一张“消息卡片 -> 信号波 -> 终端”示意视觉。
4. 底部按钮可直接跳转到发送页。
5. 与 SSE 连接状态联动，连接断开时整体 tone 会转为 warning。

它当前承担的是入口与状态提示作用，不是复杂总览 dashboard。

### 5.2 发消息页 Send

当前 Send 页是两阶段发送流：

| 阶段 | 作用 |
|---|---|
| target | 选目标：设备、部门、全员 |
| compose | 选消息级别并填写标题与正文 |

当前已确认能力：

1. 设备模式可多选设备并搜索设备、设备别名、部门。
2. 部门模式可搜索现有部门值。
3. 全员模式展示当前设备总量与在线量。
4. 消息级别有 info、notice、warn、emergency 四种。
5. warn 与 emergency 在发送 payload 中会映射到 acknowledged 模式。
6. 多设备单发场景会逐台调用发送接口，而不是由前端自己拼一个设备数组给后端。
7. 发送成功后，单目标会跳消息详情，多目标会回历史页。

### 5.3 历史页 Messages

当前历史页提供：

1. 最近消息列表读取，默认 limit 120。
2. 标题、创建时间、状态、目标的列表展示。
3. 关键字搜索，覆盖 msg_id、标题、目标、状态文本。
4. 点击任意一项进入历史详情页。
5. 对 message.created、message.status_change、message.recipient_update、message.event 四类 SSE 事件自动刷新。

### 5.4 设备页 Devices

当前设备页提供：

1. 设备列表读取。
2. 以 dept 作为二级筛选标签。
3. 展示设备别名、device_id、部门、在线状态。
4. 点击进入设备详情页。
5. 对 device.online、device.offline、device.updated、batch.revoked、batch.restored、batch.deleted 自动刷新。

### 5.5 设备详情页 DeviceDetail

当前设备详情页已经形成一个比较完整的设备观察面：

| 区块 | 当前内容 |
|---|---|
| 顶部身份 | device_id |
| 在线快照 | 在线/离线、RSSI、电量 |
| 归属信息 | 部门、批次 |
| 固件信息 | fw_version |
| 最近上报 | last_seen_at |
| 行为图 | DeviceBehaviorPanel 内联展示 |
| 可编辑字段 | alias、dept |
| 消息列表 | 最近 10 条消息 |

当前页面还具备以下特征：

1. 详情页加载器会同时取设备详情和最近 10 条消息。
2. 如果所在批次被删除，页面会自动跳回设备列表。
3. 会根据 message.created 的 target_kind 自动判断该消息是否与当前设备有关。
4. alias 与 dept 支持原地保存。

### 5.6 批次页 Batches

当前批次页提供：

1. 全部、有效、已撤销三种状态切换。
2. 列表中显示 batch_uuid、别名、状态、在线数/计划数。
3. 新建批次模式下可填写 batch_uuid、alias、produced_count。
4. 创建成功后会在页面上展示新生成的 batch_secret。
5. 对 batch.created、batch.updated、batch.revoked、batch.restored、batch.deleted 以及设备在线变化自动刷新。

### 5.7 设置页

当前设置页实际包括以下内容：

| 子页 | 当前作用 |
|---|---|
| Admins | 管理员列表、新建管理员 |
| AdminDetail | 管理员单项详情与修改 |
| Audit | 审计日志列表与展开详情 |
| About | 关于说明 |

Admins 页当前已确认能力：

1. 仅 admin 可见和可用。
2. 可创建 operator、admin、viewer 三类账户。
3. 列表中展示用户名、角色、启停状态、最后登录时间。
4. 监听 admin.created、admin.updated、admin.deleted、admin.login 事件自动刷新。

Audit 页当前已确认能力：

1. 默认读取最近 120 条审计。
2. 行内显示 actor、action、target。
3. 点击可展开时间、actor_type、IP、detail_json。
4. 监听 audit.created 事件自动刷新。

## 6. 当前实时刷新机制

### 6.1 全局 SSE store

前端没有在每个页面独立维护 EventSource，而是统一由 stream store 管理：

1. EventSource 指向 /beacon/api/v1/stream。
2. withCredentials=true，沿用 Cookie 会话。
3. 收到 hello 事件后把 connected 置为 true，并把重连延迟重置为 1 秒。
4. 断线后用指数退避重连，最大 30 秒。
5. 支持按事件名订阅，也支持通配订阅 *。

### 6.2 useAsyncLoader 的实际作用

useAsyncLoader 已经成为前端局部数据刷新的统一机制，支持：

1. onMounted 自动首刷。
2. watchSources 触发 reload。
3. streamEvents 触发 reload。
4. streamFilter 对具体事件做命中判断。
5. streamDebounceMs 控制事件风暴时的合并刷新。
6. 失败后通过 Element Plus 弹错误消息。

这说明当前控制台的“实时刷新”不是分散手写，而是已经形成一套共用模式。

## 7. 当前设备行为图的真实状态

### 7.1 当前图表位置与样式结论

设备行为图目前位于设备详情页顶部信息区中，“最后上报”下一行，采用简约内联样式，而不是卡片式大块容器。

### 7.2 当前图表窗口配置

| 窗口 | 当前 bucket |
|---|---|
| 1H | 60 秒 |
| 3H | 180 秒 |
| 6H | 300 秒 |
| 12H | 600 秒 |
| 24H | 1200 秒 |

### 7.3 当前图表系列定义

| 系列 | 数据来源字段 | 当前前端文案 |
|---|---|---|
| static_s | 静止秒数 | 静止 |
| walk_slow_s | 低速活动秒数 | 低幅活动 |
| walk_fast_s | 中速活动秒数 | 中幅活动 |
| run_s | 高强度活动秒数 | 高幅活动 |
| shake_or_fall_s | 剧烈波动秒数 | 剧烈波动 |

### 7.4 当前交互特征

1. 左右箭头支持把当前观察窗口向前或向后滚动半个窗口宽度。
2. 如果当前窗口处于实时尾部，收到 device.behavior 事件后会自动刷新并对齐到最新时间。
3. Tooltip 采用浅底、细边、轻阴影的浮层样式。
4. Y 轴上限固定为当前 bucket 秒数，因此柱子呈现“贴边”的视觉风格。
5. 图表在没有数据、加载中、加载失败三种状态下都有独立文案。

### 7.5 当前图表与后端的边界

当前 DeviceBehaviorPanel 单独请求 devicesApi.behavior，不和 DeviceDetail 主详情共用一个大接口。这意味着：

1. 行为图可以独立自动刷新。
2. 即使详情主信息未变化，行为图也可单独拉新。
3. 设备详情页没有被塞成一个难以维护的超大 loader。

## 8. 当前前端与旧规格的差异结论

### 8.1 已确认的差异

| 旧描述 | 当前真实实现 |
|---|---|
| 首页以“发消息”为主工作台 | 当前首页是轻量介绍入口，发送在独立页完成 |
| 顶层 5 项 + 助手页 | 当前为 home、send、history、devices、batches、settings，没有助手页 |
| 复杂多助手或多工作台布局 | 当前是一套轻量单控制台结构 |
| 以 Chat 为关键交互面 | 当前以消息发送、设备详情、批次管理为核心 |

### 8.2 当前应如何准确描述前端

当前前端最准确的描述应是：

“一个面向 Beacon 终端的消息发布与设备管理控制台，重点页面为发送、历史、设备、批次与设置，实时更新依赖 SSE，设备详情支持行为时间轴可视化。”

## 9. 前端完工结论

当前 BeaconOps 前端已经不是只展示静态表单的壳层，而是具备以下完整能力：

1. 能完成管理员登录与角色分权。
2. 能完成单设备、部门、全员三个维度的消息发送。
3. 能实时查看消息历史与状态变化。
4. 能查看设备列表、设备详情、近期消息和设备行为图。
5. 能查看批次状态、创建批次并拿到新密钥。
6. 能管理管理员账户并查看审计日志。
7. 能通过统一 SSE + useAsyncLoader 机制保持界面和后台状态同步。

因此，前端已达到“可用于日常运营和设备观察”的控制台完工状态，而不是设计稿阶段。
