## 4. 固件启动链路

### 4.1 当前真实启动顺序

根据 main.cpp，当前固件的装配顺序如下：

| 顺序 | 阶段 | 主要动作 |
|---|---|---|
| 1 | 基础硬件 | GPIO、I2C、传感器、LVGL、背光、NVS、settings、SPIFFS、audio、CW2017 |
| 2 | 基础业务 | error、notify、UI |
| 3 | Wi-Fi 句柄 | 创建 Wi-Fi 管理句柄，但不在此处阻塞连接 |
| 4 | net_bringup_task | 启动网络拉起任务 |
| 5 | Wi-Fi 连接 | 阻塞等待 Wi-Fi 连接成功 |
| 6 | DNS 覆盖 | 接入后写入主备 DNS |
| 7 | 省电模式 | 启用 Wi-Fi modem-sleep |
| 8 | SNTP | 阻塞等待时间同步成功 |
| 9 | 设备身份 | 读取 eFuse MAC，生成 device_id，加载批次标识 |
| 10 | event 组件 | 在 MQTT 之前先创建 event，上线前异常可短暂缓存 |
| 11 | MQTT | 用 batch_uuid + 动态 HMAC password 建立连接 |
| 12 | tx / msg / profile / health | 启动 ACK 重发、消息调度、行为窗口、健康上报 |

### 4.2 启动链路的关键结论

1. SNTP 成功是 MQTT 连接前置条件，因为 HMAC password 依赖当前时间。
2. 设备身份不是烧录唯一 UUID，而是运行时从 eFuse MAC 读取的 12 位十六进制字符串。
3. 批次凭证是编译期常量，不走设备端用户输入。
4. event 组件早于 MQTT 初始化，说明设备端异常上报被视为一等能力，而不是附属日志。

## 5. 设备身份与接入方式

### 5.1 当前身份模型

| 项 | 当前实现 |
|---|---|
| device_id 来源 | 运行时读取 eFuse MAC，转 12 位小写十六进制 |
| batch_uuid 来源 | 编译期常量 |
| batch_secret 来源 | 编译期常量 |
| MQTT username | batch_uuid |
| MQTT password | 由当前 device_id、当前 unix 时间、随机 nonce 生成的 HMAC 字符串 |

### 5.2 当前 HMAC 口令构成结论

当前 password 是三段式字符串：

1. 当前 unix 时间
2. 随机 nonce
3. HMAC-SHA256 摘要

### 5.3 当前身份链路的工程含义

1. 设备唯一身份来自芯片 MAC，因此不需要逐台写唯一业务 ID。
2. 批次授权来自固件常量，因此“切批次”的当前手段是重新编译并烧录固件。
3. 时间不同步时不允许开始 MQTT 业务连接，减少鉴权失败和重放误判。

## 7. 当前设备端消息处理链路

### 7.1 下行消息接收链

当前设备端消息链路可以确认如下：

1. MQTT 客户端订阅单播 topic、全员广播 topic，以及可选的部门广播 topic。
2. 收到 payload 后，通过 cb_mqtt_msg 转入 msg_ingest。
3. parser 只接受扁平 schema：id、ts、ttl、level、title、body、ack。
4. level 非法、正文与标题同时为空、ack 非法时会被拒收。
5. warn 与 emergency 会在 parser 阶段被强制改为 acknowledged 模式，且 ttl 视为永不过期。

### 7.2 设备端 ACK 节点

当前设备端支持四个 ACK 节点：

| ACK 名称 | 含义 |
|---|---|
| received | 已落盘或已入本地队列 |
| displayed | 首次上屏成功 |
| acknowledged | 用户摇一摇确认 |
| expired | TTL 到期但未确认 |

### 7.3 ACK 发送机制

| 项 | 当前实现 |
|---|---|
| 持久化位置 | NVS 独立命名空间 tx_pending |
| 后台任务 | tx_drain task |
| 默认最大积压 | 32 条 |
| 默认每轮发送数 | 4 条 |
| 默认重试周期 | 5000 ms |
| 默认最大重试次数 | 10 次 |
| QoS 特征 | acknowledged 优先级最高，其他 ACK 较低 |

### 7.4 ACK 失败后的处理

ACK 连续失败达到上限后，设备端会：

1. 将该 ACK 标记为失败。
2. 通过回调上报 ack_give_up 事件。
3. 把事件交给 event 组件走 uplink/event 路径。

### 8.2 行为窗口上报

| 项 | 当前实现 |

| 发送触发 | profile timer task 周期发布 |

### 8.3 健康上报

| 项 | 当前实现 |
|---|---|
| 默认周期 | 30 秒 |
| 额外触发 | 电量变化至少 3%、RSSI 变化至少 10dB，且满足 5 秒去抖 |
| 上报内容 | fw、batch、ts、电量、充电、RSSI、IP、uptime、spiffs_pending、ack_pending、drop_count |
| 上报通道 | device/{id}/uplink/health |

### 8.4 设备端异常事件

| 事件 | 当前意义 |
|---|---|
| ack_give_up | ACK 达到上限仍发送失败 |
| parse_reject | 下行消息 payload 不合法 |
| store_full | 本地存储满 |

## 10. 当前固件的已确认限制

### 10.1 部门广播订阅尚未真正装配

虽然协议和 MQTT 组件支持部门广播 topic，但在 main.cpp 中 MQTT 配置的 dept 仍为 TODO 状态，因此当前固件运行时只明确订阅：

1. device/{id}/cmd
2. broadcast/all/cmd

部门广播路径在服务器和协议层存在，但当前固件装配未完成闭环。

### 10.2 parser.h 与 parser.c 的说明不完全一致

当前 parser.c 已严格切换到扁平协议并把 audio_text 置空，但 parser.h 顶部说明仍保留旧字段注释。这属于注释滞后，不影响实际运行逻辑。

### 10.3 音频驱动存在，但 TTS 业务链已移除

固件仍初始化 audio 和 notify，但当前消息链路不再通过 on_tts 播放远端音频，因此现阶段音频能力主要用于本地提示音，而不是语音播报业务。

### 10.4 凭证与 Wi-Fi 仍属编译期配置

当前设备端没有面向现场用户的交互式配网入口；批次凭证与 Wi-Fi 列表均属于固件构建时写入。

## 11. 硬件与固件完工结论

当前 BeaconOps 固件已经完成以下闭环：

1. 设备上电后自动接入 Wi-Fi。
2. 时间同步后使用 MAC + 批次凭证完成 MQTT 鉴权。
3. 服务器可以下发扁平消息 payload，设备可以严格解析并入队。
4. 设备可以在本地进行显示、摇一摇确认、ACK 持久重发。
5. 设备可以周期上报 health 与 profile，两条链路分别服务于运维状态与行为数据。
6. 前端“今日行为图”所用数据，已经有真实端侧来源，不是模拟数据。

因此，从固件角度看，BeaconOps 已经不是概念样机，而是具备消息接收、状态观测、行为回传三条主线的可运行系统。
