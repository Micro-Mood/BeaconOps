## 2. 当前服务器的总体结构

### 2.1 当前后端技术形态

| 维度 | 当前实现 |
|---|---|
| Web 框架 | FastAPI |
| 应用版本 | 1.0.0 |
| 数据库 | SQLite，开启 WAL |
| MQTT Bridge | gmqtt 客户端 |
| 实时推送 | SSE |
| 静态控制台挂载 | /beacon/console |
| 健康检查 | /beacon/health |
| API 前缀 | /beacon/api/v1 |

### 2.2 当前启动生命周期

根据 main.py，服务启动时按以下顺序工作：

1. 初始化数据目录与日志。
2. 初始化 SQLite 并执行 schema/migration。
3. 向 MQTT 客户端注入 uplink_handler.dispatch。
4. 尝试启动 gmqtt 连接。
5. 启动消息重试后台任务 retry_loop。
6. 启动 nonce 清理后台任务 cleanup_loop。
7. 对外暴露业务 API、SSE、健康检查和控制台静态资源。

### 2.3 当前后台任务

| 任务 | 当前职责 | 周期或行为 |
|---|---|---|
| retry_loop | 扫描需要发送或重试的下行消息 | 每 1 秒 |
| cleanup_loop | 清理过期 mqtt_nonces | 每 600 秒 |
| MQTT 连接 | 订阅设备上行主题并转交业务处理 | 持续运行 |

## 3. 当前认证与授权模型

### 3.1 管理员认证

| 项 | 当前实现 |
|---|---|
| 登录接口 | POST /beacon/api/v1/auth/login |
| 登录载体 | Cookie 会话 |
| Session Cookie 名 | beacon_session |
| CSRF Cookie 名 | beacon_csrf |
| CSRF Header 名 | x-csrf-token |
| JWT 算法 | HS256 |
| JWT 有效期 | 8 小时 |

### 3.2 管理员角色

| 角色 | 当前权限结论 |
|---|---|
| admin | 可管理批次、设备、消息、管理员、审计 |
| operator | 可发送消息、编辑批次、编辑设备、旋转批次密钥、撤销/恢复批次 |
| viewer | 只读访问；获取批次详情时不返回 batch_secret |

### 3.3 CSRF 策略

当前服务只对以下情况执行 CSRF 校验：

1. 方法属于 POST、PUT、PATCH、DELETE。
2. 路径在 /beacon/api/v1 下。
3. 不属于 auth/login 与 auth/device。
4. 不是 Bearer Token 形式调用。

这说明当前控制台和后台 API 是典型的 Cookie + CSRF 保护模式，而不是纯 JWT Header 模式。

## 4. 设备接入鉴权模型

### 4.1 当前接入凭证模型

| 字段 | 当前结论 |
|---|---|
| MQTT client_id | device_id |
| MQTT username | batch_uuid |
| MQTT password | ts:nonce:hmac_hex |
| HMAC 算法 | HMAC-SHA256 |
| 服务器允许时间偏差 | 300 秒 |
| nonce 过期时间 | 600 秒 |

### 4.2 当前鉴权校验点

服务端在 identity.verify_connect 中确认以下条件：

1. client_id 必须是 12 位小写十六进制。
2. batch_uuid 必须存在。
3. password 必须是三段式。
4. ts 必须在允许时钟偏差内。
5. batch 必须存在且未撤销。
6. 设备如已注册则必须 enabled=1。
7. 首次接入时，当前批次设备数不得超过 produced_count。
8. HMAC 必须匹配 batch_secret。
9. nonce 必须未被使用过。

### 4.3 当前鉴权成功后的动作

| 场景 | 当前行为 |
|---|---|
| 首次见到设备 | 自动插入 devices 记录 |
| 已存在设备重新接入 | 更新 batch_uuid、last_seen_at、updated_at |
| 设备换批次重连 | 服务端会修正设备的 batch_uuid |

### 4.4 当前 ACL 结论

设备接入成功后，服务端构造以下 ACL：

| 方向 | topic |
|---|---|
| 订阅 | device/{device_id}/cmd |
| 订阅 | broadcast/all/cmd |
| 订阅 | broadcast/dept/{dept}/cmd，只有设备有 dept 时才存在 |
| 发布 | device/{device_id}/status |
| 发布 | device/{device_id}/uplink/ack |
| 发布 | device/{device_id}/uplink/event |
| 发布 | device/{device_id}/uplink/health |
| 发布 | device/{device_id}/uplink/profile |

## 7. 当前消息投递模型

### 7.1 当前消息输入约束

POST /messages 当前接受以下业务字段：

| 字段 | 当前约束 |
|---|---|
| target_kind | device、all、dept |
| target_value | 目标设备号或部门值；全员发送时可为空 |
| level | info、notice、warn、emergency |
| title / body | 不能同时为空 |
| ttl | 默认 3600 秒 |
| ack_mode | none、received、displayed、acknowledged |

### 7.2 当前服务端的强制业务规则

1. warn 与 emergency 自动强制为 acknowledged 模式。
2. emergency 自动强制 ttl 为 0，即永不过期。
3. 目标设备不存在、已禁用、所在批次已撤销时，不允许创建有效 recipient。
4. target_kind 为 all 或 dept 时，如果当前没有任何活跃 recipient，会直接报错。

### 7.3 当前 recipient 级状态推进

| 阶段 | 含义 |
|---|---|
| queued | 已落库，等待发送 |
| sent | 已发出，但还在等待 ACK |
| delivered | 已投递完成；通常对应不要求 ACK 的消息 |
| displayed | 设备确认已显示 |
| acknowledged | 用户确认 |
| expired | TTL 到期 |
| failed | 发送失败或 ACK 超时失败 |

### 7.4 当前发送策略

| 项 | 当前实现 |
|---|---|
| 发送 topic | device/{device_id}/cmd |
| 即时发送 | 创建消息后异步立即尝试 |
| 离线补发 | 设备上线时 on_device_online 重新扫描 |
| 最大重试次数 | 10 |
| 指数退避起点 | 2 秒 |
| 退避上限 | 300 秒 |
| 扫描周期 | 1 秒 |
| 单轮批量限制 | 64 recipient，32 条消息扫描 |

### 7.5 当前消息聚合状态

messages 表中的 status 是聚合视角，而不是单设备视角。服务端通过 recipient 明细刷新聚合状态，用于前端历史页和详情页展示。

## 8. 当前上行处理模型

### 8.1 MQTT 订阅面

gmqtt 连接成功后，服务端订阅以下主题：

| topic | QoS | 用途 |
|---|---|---|
| device/+/status | 1 | 在线状态与固件版本 |
| device/+/uplink/ack | 1 | ACK 与状态推进 |
| device/+/uplink/event | 1 | 端侧异常事件 |
| device/+/uplink/health | 0 | 健康快照 |
| device/+/uplink/profile | 1 | 行为窗口数据 |

### 8.2 上行接受前置条件

所有上行在真正入库前，都要经过 _accept_uplink：

1. 设备必须已经通过鉴权并在 devices 中存在。
2. 设备必须 enabled=1。
3. 所属批次必须未 revoked。
4. 若设备已被禁用或批次被撤销，会被强制更新为 offline=0 并拒收上行。

### 8.3 五类上行的当前处理方式

| 类型 | 当前行为 |
|---|---|
| status | 更新 online、fw_version、last_seen_at，并发 device.online 或 device.offline 事件 |
| ack | 交给 messages.on_ack 推进 recipient 与 message 状态 |
| event | 交给 messages.on_device_event 记录设备侧事件 |
| health | 更新 devices 健康快照，可选写入 device_health_history，并发 device.health |
| profile | 插入 behavior 表，更新 last_seen_at，并发 device.behavior |

## 9. 当前 SSE 实时模型

### 9.1 基本实现

当前控制台实时刷新使用 SSE，而不是 WebSocket。

SSE 的核心行为如下：

1. 每个订阅者建立一个队列，默认上限 200 条。
2. 连接成功后立即发送 hello 事件。
3. 若 15 秒内没有新事件，则发送 ping 注释保活。
4. 前端收到事件后按页面需要执行局部 reload。

### 9.2 当前已经确认存在的事件类别

从路由、消息核心、批次、管理员、设备、上行处理可确认以下事件族：

| 事件族 | 例子 |
|---|---|
| 管理员 | admin.login、admin.created、admin.updated、admin.deleted |
| 批次 | batch.created、batch.updated、batch.revoked、batch.restored、batch.deleted |
| 设备 | device.online、device.offline、device.updated、device.health、device.behavior |
| 消息 | message.created、message.recipient_update、message.status_change、message.event |

## 10. 当前运维与安全边界

### 10.1 当前安全边界

1. 管理后台使用 Cookie + CSRF 防护。
2. 设备接入必须通过 HMAC、批次密钥、nonce 防重放。
3. 批次撤销后，服务端不仅更新数据库，还会异步重启 Mosquitto，使已连接设备立即失效。
4. health history 默认关闭，避免把当前系统误判为默认自带长期时序存储。

### 10.2 当前明确不是服务器现状的内容

以下内容不应写成“当前服务器已经实现”：

1. TimescaleDB 或 Redis 缓存层。
2. WebSocket 主推实时架构。
3. 独立的用户、holder、工单、知识库或 chat 服务域。
4. 面向设备的 OTA 升级 API。
5. 默认开启的 health 历史曲线后端。

## 11. 服务器完工结论

当前 BeaconOps 服务器已经具备完整的消息调度与设备管理骨架：

1. 可以管理批次、管理员、设备与审计。
2. 可以校验设备接入、控制 topic 访问与批次容量。
3. 可以按单设备、全员、部门三个维度下发消息。
4. 可以跟踪每个 recipient 的 sent、displayed、acknowledged、expired、failed 状态。
5. 可以接收 status、ack、event、health、profile 五类上行并推给前端实时刷新。
6. 可以把 profile 聚合为设备行为时间轴，供控制台图表直接消费。
